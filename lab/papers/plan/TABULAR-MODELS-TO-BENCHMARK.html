3. Systems-researcher recommendation — ONE small model to host locally

Recommended: Qwen2.5-VL-7B-Instruct (pin by HF revision/commit at selection time), served via vLLM on a single ~24 GB GPU (bf16/AWQ-int4 fallback for ≤16 GB).

Rationale (efficient-ml / frontier-smol view):

  1. Best capability/size on tables in the open tier — native HTML &

bbox-grounded cell output; strong on dense/merged finance tables in reported evaluations; the program's "SOTA-via-open-prompted" thesis rests on a VLM, and 7B is the smallest size that is competitive on hard tables without fine-tune.

  1. Deployable on commodity hardware — fits one 24 GB GPU at bf16;

int4 (AWQ/GPTQ) brings it to ~10–12 GB with modest quality loss, enabling a realistic apis.hyperbots.com self-host path.

  1. Mature serving — first-class vLLM/SGLang support, OpenAI-

compatible server → trivial handshake parity with the API spec (see TABULAR-API-HANDSHAKE.md).

  1. License — Qwen2.5-VL is Apache-2.0-class permissive (confirm exact

license text at pin time), suitable for hosted commercial inference.

Runners-up / fallbacks:

  • GOT-OCR2.0 (~0.6B) — far cheaper, CPU-feasible; pick if the latency/

cost budget dominates and table complexity is moderate (it is OCR-2.0, weaker than a 7B VLM on heavily merged finance tables).

  • InternVL3-8B — closest alternative if Qwen licensing/availability

blocks; comparable footprint.

  • MiniCPM-V-2.6 (8B) — efficient fallback with good doc grounding.

HOSTING-GAP STATEMENT (mandatory honesty): No GPU or serving infrastructure exists in this environment. The recommendation is a design selection only. 5. Per-model prompt / decoding / harness-invocation spec (DESIGNED — not run)

plan/TABULAR-BENCHMARK-WIRING.md §5 explicitly defers "concrete per-model adapter behavior" to this doc. Filling that named gap. Spec only — no model invoked, no adapter executed, no number produced. The benchmarked prompt + decoding are NOT chosen here: they are already frozen in harness/tasks.json TR-000 and are identical for every model in the tabular family. This section only specifies how each adapter wraps that one frozen artifact — chat-formatting lives ONLY in the adapter (tasks.json _note contract).

5.1 The single frozen artifact (verbatim from harness/tasks.json)

  • Prompt template (task-level, ONE for the whole family):

"Recognize the table in the document below. Output ONLY the table as valid HTML using <table>, <tr>, <td>/<th>, with rowspan/colspan for merged cells, preserving the exact cell text and reading order. Output nothing except the HTML table. Document:\n---\n{input_text}\n---\nHTML table:"

  • Decoding (task-pinned, ONE for the whole family):

temperature=0.0, top_p=1.0, max_tokens=4096, stop=[], seed=20260516.

  • No adapter may change the instruction text or the decoding values.

The ONLY per-model freedom is (a) how {input_text} is realized (text vs image) and (b) the model-native chat envelope around the unchanged instruction.

5.2 {input_text} realization by family (image-substitution hook)

Per wiring-doc §5, for family == "tabular" the adapter substitutes the table's rendered page image for the {input_text} slot while keeping the frozen instruction verbatim:

Family{input_text} realized asEnvelope the adapter builds
Open VLM (Qwen2.5-VL, InternVL3, MiniCPM-V, Phi-3.5-V, olmOCR)rendered page image (PNG, page-native DPI; no client-side crop unless the GT slice is a crop)model-native multimodal chat: [{role:user, content:[{image}, {text: <frozen instruction with the literal "{input_text}" marker removed and the image taking its place>}]}]
OCR-2.0 e2e (GOT-OCR2.0, dots.ocr, PaddleOCR-VL)rendered page imagemodel-native "OCR→HTML/markdown table" mode; frozen instruction passed where the model accepts a task string; HTML table parsed from the model's structured output

Image rendering itself is part of the (EXECUTION-PENDING) loader (tabular_loader.py, wiring-doc §1); this section only fixes the contract, it does not render anything here.

5.3 Decoding reconciliation across the three pinned surfaces

Three places independently pin decoding; they MUST agree, and the adapter is responsible for asserting it (offline, at run time):

  1. Benchmark (authoritative): tasks.json TR-000 — `temp 0.0,

top_p 1.0, max_tokens 4096, seed 20260516`.

  1. Open-weights serving (vLLM/SGLang/HF): adapter passes exactly

the §5.1 values to the engine; greedy (temp 0.0) makes top_p inert but is still sent for an explicit, logged match. Models that ignore seed under greedy → adapter records seed_effective: "n/a (greedy-deterministic)" rather than asserting a false seed match.

  1. Hosted API / apis.hyperbots.com: decoding is server-pinned and

surfaced as decoding_digest (TABULAR-API-HANDSHAKE.md §4/§7); the adapter compares that digest to the benchmark pin and fails the run (does not silently proceed) on mismatch — no cell is emitted from a decoding-divergent run.
