TABULAR REASONING — API Handshake Spec (apis.hyperbots.com)

_Owner: cto-org finance-doc-ml-engineer (= the ML engineer for the API handshake, per program mandate). Design only. This spec is NOT implemented against the live API and was NOT called from this environment. It is deliberately consistent with the existing hyperapi-sdk wire contract (X-API-Key, X-Request-ID, X-Async, presigned POST + S3 PUT, POST /v1/<op> with document_key, GET /v1/jobs/{id} poll, typed-error taxonomy) so a future tabular op drops into the same SDK with no new transport. Product accuracy claims are per apis.hyperbots.com only and appear nowhere here._

1. Operation

New op POST /v1/tabular — table structure recognition (TSR) on an already-uploaded document, returning the recognized table(s) as HTML + a normalized grid. Same submit+poll envelope as /v1/extract and /v1/split (reuses the SDK's _submit_and_wait machinery; no new client code beyond a thin tabular(document_key, ...) method and a typed result/error).

2. Auth (redacted)

  • Header: X-API-Key: hk_<redacted> (key class hk_, never logged,

never committed; SDK resolves from arg or HYPERAPI_KEY env).

  • Every request carries a client-generated X-Request-ID (UUIDv4),

echoed in responses and errors for traceability.

  • TLS only; key scoping/quotas per the existing apis.hyperbots.com

account model (unchanged by this op).

3. Upload modes (unchanged from SDK contract)

  1. Presigned flow (canonical): POST /v1/documents/upload →

{ upload_url, expires_in, document_key, ... }; client PUTs bytes to upload_url (S3); then references the returned document_key. Supports large multi-page PDFs/scans (the enterprise-table case).

  1. Inline (small images): document_key may instead be an inline

base64 payload field for single-page table crops under the size cap (cap = existing platform inline limit; reject over-cap with payload_too_large, do not silently truncate).

Accepted media: application/pdf, image/png, image/jpeg, image/tiff (multi-frame TIFF = multi-page). Page selection optional (pages: [int], 1-indexed) to scope TSR to specific pages.

4. Request schema


POST /v1/tabular
X-API-Key: hk_<redacted>
X-Request-ID: <uuid>
X-Async: true            # async submit+poll (recommended for multi-page)
Content-Type: application/json

{
  "document_key": "<key from /v1/documents/upload>",
  "pages": [1, 2],                 // optional; default = all pages
  "output": {
    "format": "html",             // "html" (canonical) | "grid" | "both"
    "include_cell_bboxes": true,  // per-cell [x0,y0,x1,y1] in page coords
    "merge_spanning": true        // emit rowspan/colspan vs flattened
  },
  "model": "tabular-default",     // pinned server-side model id (versioned)
  "options": { "max_tables_per_page": 16 }
}

Decoding for the served model is pinned server-side and surfaced in the response (model_version + decoding_digest) — the client does NOT choose temperature/top_p, mirroring the benchmark's pinned-decoding discipline (TR-000: temp 0.0, top_p 1.0, seed 20260516).

5. Response schema

Async submit → 202 { "job_id": "...", "status": "queued", "poll_url": "/v1/jobs/<id>" }. Poll GET /v1/jobs/<id> →


200
{
  "status": "succeeded",          // queued|running|succeeded|failed
  "request_id": "<uuid>",
  "model_version": "tabular-default@2026-05-20-rev<sha>",
  "decoding_digest": "sha256:<pinned-decoding-hash>",
  "result": {
    "tables": [
      {
        "page": 1,
        "table_index": 0,
        "bbox": [x0,y0,x1,y1],
        "html": "<table>…</table>",
        "grid": {                 // present if output.format includes grid
          "n_rows": 12, "n_cols": 6,
          "cells": [
            { "row": 0, "col": 0, "rowspan": 1, "colspan": 2,
              "text": "Description", "bbox": [..], "is_header": true }
          ]
        },
        "confidence": 0.0          // model-reported; advisory only, not a
                                   // benchmark metric (never a TR-000 cell)
      }
    ]
  }
}

Synchronous mode (X-Async: false, single small image): same body returned 200 inline without job_id.

7. Versioning

  • Op path stable at /v1/tabular; behavioral changes are gated by the

server-side model_version (<id>@<date>-rev<sha>) returned on every response — clients pin/observe it for reproducibility parity with the benchmark's pinned harness/models.json revisions.

  • decoding_digest lets a caller assert the served decoding equals the

benchmarked TR-000 pin (offline comparability).

  • Deprecations: additive fields only within /v1; breaking changes →

/v2/tabular, dual-served through a documented window.

8. Benchmark ↔ product separation (honesty)

  • This handshake describes how the selected/hosted tabular model is

served. The TR-000 leaderboard scores models under the pinned factory harness; a product confidence field is advisory only and is never injected into a TR-000 cell (same rule as hyperbots-ip-seed-v1 in harness/models.json).

  • No accuracy/latency number appears in this spec. Product performance

is per apis.hyperbots.com only; benchmark numbers come solely from reproduced-or-provenance-tagged TR-000 runs.
