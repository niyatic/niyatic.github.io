Audit caveats — read before citing

  1. Claude regex cells (warn badges). The original Claude-Opus and Claude-Sonnet cells (without -LLM suffix) were produced by sub-agents that fell back to PyMuPDF + Claude-authored regex/heuristics instead of doing per-doc Claude inference. They are labelled inference_method:claude_authored_regex_on_pymupdf_text in the cell JSON. Treat as apples-to-oranges vs the -LLM rows.
  2. Claude Sonnet LLM read text, not pixels. The pdftoppm dependency was broken on the host; Sonnet's sub-agent fell back to reading the regex-extracted text. The 0.806 extract F1 is real LLM reasoning on text, not visual LLM on the PDF.
  3. Claude Opus LLM over-redacted predictions. Interpreted the "no customer-name externalization" rule as redacting account_number from prediction JSON — that's overreach (the rule is about stdout/logs, not cell content). Extract precision tanked as a result. The line-items count exact-match (0.886) is the cleaner signal.
  4. Claude Haiku LLM partial. Socket error mid-run (33 tool uses in). Extract emitted 0; classify + split partial. Re-dispatch needed for clean Haiku row.
  5. Gemini-FL leads extract on a clean methodology. 0.5995 micro-F1 with PDF-native input, no privacy-redaction overreach, no fallback text. Headline-quality cell.
  6. HyperAPI extract scored 0.063 because the extract entities have different field names (e.g. receiver_details, ship_from_details) than the tough gold schema. The cell needs a field-name remapping before any product-accuracy claim is drawn from it; until then, all HyperAPI product claims are per apis.hyperbots.com verbatim.
