3.1 Speed

3.2 Depth — the formal contract

Lifted directly from docs/IDEMPOTENCY.md:

"For a fixed connector + fixed config, replaying any subset of
already-processed sources produces zero net change in the target and a
byte-identical emit artifact. A re-seen idempotency key is a successful
skip recorded in skipped_dup — never an error, never a second write."

The post unpacks this in five movements:

1. Byte-hash default is the safe default. SourceBytesKey = SHA-256

over source bytes, optionally salted by target scope. Same source bytes

in => same key => no-op skip. Re-issued credit memo arrives as new

bytes and is correctly processed — never silently skipped.

2. BusinessKey is the opt-in override. Deterministic composite over

target-scoped business fields. Used where the target enforces its own

uniqueness rule.

3. Two lines of dedup. Framework SeenKeyStore + the target's own

duplicate rejection. Both resolve to skipped_dup. Belt + braces.

4. check_replay_safety is non-optional. The conformance kit fails

the build if replay isn't a no-op + byte-identical.

5. The exit-code gate. count(sources) == sum(counts(sinks)) or the

process exits non-zero. Idempotency is a property of the *gate*, not

just the code.

Code excerpts from the byte-exact sample in README.md:

sink_counts: {'posted': 2, 'dead_letter': 0, 'unresolved': 0,
              'skipped_dup': 0, 'failed': 0, 'audit': 3}
invariant_holds: True
exit_code: 0

Re-run on the same sources:

sink_counts: {'posted': 0, 'dead_letter': 0, 'unresolved': 0,
              'skipped_dup': 2, 'failed': 0, 'audit': 3}
invariant_holds: True
exit_code: 0

Byte-identical emit artifact across the two runs — that is what the

replay contract means in production.
