Schema drift in production: what happens when a customer adds a custom field — plan

1. Headline + hook

Hook: **"On Monday morning, the customer's NetSuite admin adds a custom

segment field to vendor bills. The bespoke integration breaks silently.

The connector framework's schema_version header catches it; the affected

records dead-letter with the raw response attached; nothing is silently

posted with a missing field. Here is how that contract works."**

The post is the *operational reality* counterpart to blog #5 — drift is

the failure mode the framework is shaped around.

2. Audience + intent

  • Primary: integration engineers + change-management owners.
  • Reader takeaway: *"schema drift is a connector-framework first-class

  • CTA: try a drift simulation on a sandbox.

3. The two-layer pitch (REQUIRED)

3.1 Speed

  • Agent re-runs the connector after the new field lands; affected records

hit dead_letter with the raw response.

  • Agent updates the normalizer to map the new field (one function); ships

the new descriptor with bumped schema_version.

  • *[PLACEHOLDER, ~1-2 hours per drift event] vs bespoke baseline often

[PLACEHOLDER, days, because schema breakage is silent].*

3.2 Depth

  • schema_version is on the descriptor. It is stamped on every

record (docs/USAGE.md + descriptor code). The CSV emit carries a

# schema_version=2.3.0 header (see byte-exact sample in

README.md).

  • Additive fields don't break the contract. The normalizer's job is

to map; an unknown additive field is recorded in lineage, not posted

blind.

  • Subtractive / type-change drift dead-letters. When a field the

normalizer expects is missing or shape-changed, the record goes to

dead_letter with the raw response attached. Never to posted.

  • Bump the schema_version, ship a new descriptor. Old records with

old schema_version continue to flow; new field is mapped going

forward. The two co-exist.

  • Lineage stamp carries schema_version on every record (per

v0.4 README — "lineage stamp on every result carries the connector's

descriptor policy").
