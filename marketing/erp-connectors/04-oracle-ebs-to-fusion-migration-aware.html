3. The two-layer pitch (REQUIRED)

3.1 Speed (claim deferred)

The speed claim is deferred until an Oracle descriptor exists. The

post can still pitch the *migration-aware* speed angle — "one connector

spans both back-ends, the cutover is configuration not re-integration" —

without naming a wall-clock number.

3.2 Depth — the migration-aware pattern

  • One descriptor family, two targets (oracle_ebs, oracle_fusion).
  • Vendor-id continuity — master_key_resolver maps EBS vendor_id

→ Fusion Supplier.SupplierId, with a per-tenant lookup table during

cutover.

  • Idempotency across the cutover — BusinessKey with the *business

invoice number*, salted by tenant + posting calendar, so a bill posted

in EBS pre-cutover and replayed post-cutover is a skipped_dup, not a

double-post.

  • Sign / FX policy aligned across back-ends — Fusion's CreditMemo

sign vs EBS's.

  • Two-line dedup — the framework's first line + each Oracle

back-end's second line (the database's unique constraint on

invoice_num).
