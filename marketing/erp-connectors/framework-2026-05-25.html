1. The two-layer thesis (the CEO directive, decoded)

The CEO directive is verbatim:

"erp-connector: for this agent also create the blogs by gtm-content-team,
use **deep connector expertise as a pitch but focus on the speed to integrate
in the agent story**."

That sentence is two layers stacked. Every blog in the set must hold both:

Layer 1 — Depth (the credibility floor)

The connector expertise that earns the right to make the speed claim. Drawn

straight from hyperapi-connector-framework/ and its docs.

  • Connector contract — Source → Extract → Normalize → Idempotent Sink.

Source src/hyperapi_connector/connector.py.

  • Descriptor policy — master-key resolution, sign convention, FX policy,

grain, schema_version, ToS/rate posture. Source

src/hyperapi_connector/descriptor.py.

  • Idempotency contract — byte-hash default (SourceBytesKey = SHA-256

over source bytes, optionally salted by target scope); BusinessKey

override for cross-file economic-event dedup. Source docs/IDEMPOTENCY.md.

  • Replay-safety formal statement — "replaying any subset of

already-processed sources produces zero net change in the target and a

byte-identical emit artifact." Verified by check_replay_safety in the

conformance kit. Source docs/IDEMPOTENCY.md.

  • Six typed sinks — posted, dead_letter, unresolved, skipped_dup,

failed, audit. The count invariant count(sources) == sum(counts(sinks))

is gated as a non-zero exit on drift. Source

src/hyperapi_connector/result.py.

  • Secrets / token boundary — OAuth2 rotate-and-persist on every refresh

with a cross-worker write lock; invalid_grant hard-stop; env/realm

assertion at process start; redacted repr. Source

src/hyperapi_connector/secrets.py.

  • Conformance kit — replay-safety, count invariant, confidence gate,

determinism, secrets hygiene, ToS posture, lineage. Source

src/hyperapi_connector/conformance/kit.py.

  • No HITL, ever — low-confidence records dead-letter with the raw response

attached. No human review queue. (docs/LIMITS.md.)

Layer 2 — Speed (the agent wedge)

The headline framing the CEO asked for: with HyperAPI agents driving

configuration through the connector framework, integration that used to take

weeks of bespoke SI engineering compresses to hours of agent-driven

configuration. This is the wedge. The depth above is what makes the claim

survivable.

The agent's job is not to write a connector from scratch; it is to drive

the configuration surface the framework exposes:

  • Pick the descriptor (target ERP, doc_type) from the registry

(src/hyperapi_connector/connectors/erp/registry.py).

  • Resolve the master-key lookup against the tenant's data.
  • Choose the idempotency strategy (byte-hash default; BusinessKey if the

target enforces uniqueness on business fields).

  • Wire credentials through the inert TokenStore boundary.
  • Run the conformance kit; ship only when all checks pass.

That is hours of work for an agent against a framework whose contract is

already proven. The same work, bespoke, against SAP IDoc handlers or

NetSuite SuiteTalk envelopes, has historically been weeks. The blogs land

that comparison — with measured numbers, never invented ones.

5. The Hyperbots bar (what differentiates each blog)

Every plan must answer: *what does the connector framework hold that a

hand-rolled SAP/NetSuite/etc. integration usually does not*?

The shipped framework answers (paths cited):

1. Replay is a contract, not a hope. Re-running on the same source bytes

posts 0, skips all, and emits a byte-identical artifact. Verified by

check_replay_safety in conformance/kit.py. The default key is byte-hash

because that is the safe default — never silently skipping a legitimate

re-issue, never silently double-posting the same file.

2. The count invariant is the gate. count(sources) == sum(counts(sinks))

or the process exits non-zero. Bespoke connectors typically log; the

framework gates.

3. **Six typed sinks (posted, dead_letter, unresolved, skipped_dup,

failed, audit).** "Failed" is the smallest of the six. The most common

bespoke-connector failure mode (silent partial posting) is not reachable in

the framework.

4. No HITL. Low-confidence records dead-letter with the raw response

attached, never a human queue. Mentioned in every blog that touches

confidence behavior; never softened.

5. Secrets boundary is inert by default. No fabricated credentials, no

logged token, no payload-embedded secret. OAuth2 rotation honors a

cross-worker write lock; invalid_grant is a hard stop, not a retry.

6. **A new connector = a ConnectorDescriptor + a Normalizer.map + a sink

choice.** No SDK edit. This is the *agent-time-compression* mechanism; the

blogs explain why this is true at the contract level, not just the demo

level.

These six bullets are on-disk-backed. Any blog claim outside this set

needs either a new on-disk artifact or a [PLACEHOLDER — depth claim needs

source] tag.

6. The honesty discipline (publish gate)

Before any blog in this set publishes externally:

1. Speed numbers are placeholders until measured. Every "X hours vs Y

weeks" headline needs either a named customer rollout (with consent;

see [[agent10-no-broadcast]] — internal-only stack-fit testing must not

leak) or a documented internal benchmark on a clean repo clone.

2. Customer names are consent-gated. ERP product names ("SAP S/4HANA",

"NetSuite", "QuickBooks Online") are fine — they are products. Customer

names are not, ever, without explicit written consent.

3. Competitor mentions need a real fetched source. Until §4 is rebuilt

from real fetches, no shipped blog names Workato / MuleSoft / Boomi / Tray

/ n8n / Make in body copy. They may be referenced in the plans for

positioning.

4. Depth claims need on-disk backing. Anything not in §5 needs either an

added artifact in hyperapi-connector-framework/ or a [PLACEHOLDER —

depth claim needs source] tag and a measurement task.

5. Don't promise completeness Hyperbots doesn't have. The 8 ERPs above

ship as compiled descriptors; the two runnable references are

invoices_to_csv and bank_txns_to_kyriba_json. Any "full-cycle ERP"

plan must check source state at publish time, not assume parity.

6. The flap rule applies. Per [[dont-die-on-flap-close]] — a blocked

measurement or a denied URL fetch is logged honestly, not silently

skipped. This framework already records the WebFetch denial in §4.

7. No cross-channel leakage. Per [[cfo-nanoclaw-channel-only]] — any

cfo-touchpoint snippet posts only to cfo-nanoclaw-channel, never to

general Teams/Slack or other product channels.

8. Hyperbots framing on the model. "Open-source model fine-tuned by

Hyperbots; fine-tune corpus, labeling, eval set, and weights are

Hyperbots IP." Never claim closed-source ownership of open weights.
