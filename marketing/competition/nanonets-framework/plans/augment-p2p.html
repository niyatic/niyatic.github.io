Augment (P2P) — deep-study plan

1. Executive summary (TEMPLATE — numbers placeholder)

Augment enriches extracted P2P fields by external lookups: vendor master (canonicalise vendor name → vendor ID + payment terms), tax- registry (validate VAT/GSTIN/EIN → jurisdiction + active status), PO master (line catalogue → expected price + UoM), FX rates (per pinned snapshot), holiday calendars (for due-date sanity), and chart-of- accounts (suggest GL account from line category). Output = enriched field set + per-enrichment confidence + provenance. Numbers [PENDING-MEASUREMENT].

2. Scope & coverage

  • Workload. Post-extract P2P documents, enriched against the lookup

surfaces above.

  • Platforms. Hyperbots augment (or sub-services: vendor-resolver,

tax-id-validator, etc.) + ≥1 competitor.

  • Coverage dims. 7 of 14.
  • Test surface.
  • n = [PLACEHOLDER] per dim.
  • Stratification: enrichment family (vendor / tax-ID / PO-line /

FX / holiday / GL), match-difficulty band (exact / fuzzy / no-match), source freshness band (current / 1-week / 1-month stale).

  • Gold: master-data-validated mappings.

3. Methodology

3.1 Data-status representation

Eight-row legend. Per-enrichment scoring: each enrichment is its own unit.

3.2 Sample design

n = [PLACEHOLDER] per dim. Master-data snapshot pinned by sha256.

3.3 Metrics

  • Per-enrichment-family P/R/F1.
  • Enrichment hit-rate = fraction of fields that get a non-null

enrichment.

  • Lookup precision = fraction of non-null enrichments that are

correct.

  • Freshness lag = wall-clock age of the master-data snapshot at

enrichment time (reported, not optimised).

  • FX-error rate = |applied FX − GT FX| distribution, with the

pinned snapshot date.

  • Provenance traceability = fraction of enrichments that cite the

source row in master data.

  • 95% bootstrap CI + n.

3.4 Scoring rubric

  • Vendor canonicalisation: casefold + whitespace-collapsed + alias

table; exact-match on canonical ID after lookup.

  • Tax-ID: format-pattern + checksum (per-jurisdiction rule) + active-

status flag.

  • FX: numeric-tolerance max([PLACEHOLDER]% rel, [PLACEHOLDER] abs)

against pinned snapshot.

  • Wrong enrichment = FP AND FN.

3.5 Evaluation controls

Frozen sha256 (data + master). Rerunnable command:


run --study augment-p2p-v0 \
    --dimension D<id> \
    --system hyperbots-idp@<model-revision> \
    --system <competitor>@<api-version> \
    --dataset p2p-augment-bench-v1@<snapshot-sha256> \
    --master-snapshot <master-sha256> \
    --split test --seed <pinned> \
    --metric enrichment_hit_rate --bootstrap 1000

3.6 Independence and disclosure

Hyperbots-published; affiliation declared. Special note: master-data snapshot is the same for all systems under test; competitor lookups that reach a different master are flagged as not-apples-to-apples.

3.7 Limitations

  • Vendor master is customer-specific; the bench uses a canonical

10k-vendor master pinned in PINS.txt.

  • Tax-registry APIs differ per jurisdiction; the bench limits to

[PLACEHOLDER — jurisdictions] with pinned validation rules.

  • Holiday calendars are region-specific; pinned per region.

3.8 Where this primitive sits in the IDP pipeline

The augment stage, downstream of extract, upstream of validate and match.
