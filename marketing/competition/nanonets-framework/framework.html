0.1 The nine-section spine

The di-study is structured around nine numbered sections, each non-optional for any post claiming this depth bar:

  1. Executive summary — headline framing + "what this study answers" + the

honest data-status disclosure.

  1. Scope & coverage — workload, platforms assessed, dimensions covered,

test surface (n, stratification, gold provenance), boundaries of the edition.

  1. Methodology — eight sub-sections (3.1–3.8) covering data-status

legend, sample design, metrics, scoring rubric, evaluation controls, independence/disclosure, limitations, and where this study sits in the IDP pipeline.

  1. Dimension analysis — for each in-scope dimension, a row carrying

ID | name | definition | measurement method | system-A result | system-B result | Δ, plus reading rules for the Δ column (non-overlapping 95% CIs only, otherwise tie-band).

  1. Combination matrix — a doc-type × failure-mode cross-tab whose rows

are strata, each tagged with the D-dimensions it exercises. The di-study ships 55 strata.

  1. Metric-taxonomy crosswalk — map the framework's dimensions onto two

external taxonomies (the 200-item complexity taxonomy + the 204-row extraction inventory) and identify the gaps the framework introduces versus existing industry practice.

  1. Hyperbots capability coverage — per HyperAPI capability cluster,

counts of dimensions covered fully / partially / not yet, sourced from the Hyperbots primitive dimensions file (NOT from measurement).

  1. Competitor offerings inventory — per platform: which named

product/SKU/version handles this primitive, with marketing source link.

  1. Findings & outlook — once measured: per-dimension winners with CI

overlap rules; outlook; recommendations for evaluators; sources.

0.3 The methodology block (verbatim shape from di-study §3)

Every deep-study post in this library must carry the same eight methodology sub-sections, only the contents change per primitive×domain:

  • 3.1 Data-status representation — Correct / Incorrect / Missed /

Spurious / Assessment in progress / Vendor-reported. No cell carries a score until that cell is measured + pinned + rerunnable. A status label is a statement about data availability; it is never a proxy for a result.

  • 3.2 Sample design — design parameter n = [PLACEHOLDER per cell]

tagged documents per dimension, stratified across the dimension's sub-categories, with macro-averaging across strata so rare-but-important categories are not outweighed by common ones. Inter-annotator agreement reported via Cohen's κ.

  • 3.3 Metrics — field-level P/R/F1 macro-averaged across the field

schema; layout-aware accuracy; structure fidelity (cell-P/R + TEDS); reporting convention: point estimate + 95% bootstrap CI + explicit n; overlapping CIs → tie-band, never a strict ranking.

  • 3.4 Scoring rubric — exact-match for identifiers/dates;

numeric-tolerance match for monetary fields (max(1% rel, 0.01 abs), pinned per cell); structure scored independently for tables; double-annotated GT with a third-reviewer adjudicator.

  • 3.5 Evaluation controls — frozen dev/test split at dataset

registration; test split never used for prompt/model tuning; equal documented training budget if any custom model is fine-tuned; contamination review per document (source, license, public date, n-gram overlap); every cell tied to a single rerunnable command: `` run --study <study-id> \ --dimension D<id> \ --system hyperbots-idp@<model-revision> \ --system <competitor>@<api-version> \ --dataset <bench-id>@<snapshot-sha256> \ --split test --seed <pinned> \ --metric <metric-id> --bootstrap 1000 ``

  • 3.6 Independence and disclosure — Hyperbots-published; affiliation

declared prominently; identical protocol/metrics/scoring for both platforms; competitor assessed using GA offerings without deliberate misconfiguration; no measured comparative figure ships without independent methodology review.

  • 3.7 Limitations — affiliation, managed-service drift (competitor API

version freezes), corpus representativeness, residual GT ambiguity, and per-primitive limits (e.g. fuzzy ground truth for reason).

  • 3.8 Where this primitive sits in the IDP pipeline — the canonical

flow is: parse → split → classify → extract → augment → validate → match → GL post → reason. Each cell explicitly scopes to one stage and declares which upstream/downstream stages it does not measure.

0.5 The Δ-column reading rule

  • A Δ with non-overlapping 95% CIs → one platform is higher on that

dimension's canonical metric.

  • A Δ whose CI includes zero → not a meaningful difference; report as

tie-band, not a strict ranking.

2. The 14-cell matrix (7 primitives × 2 domains)

2.1 The two domains

  • P2P (procure-to-pay). Document families: purchase order (PO), goods-

receipt note (GRN), supplier invoice, bill-of-lading (BoL), packing slip, credit note, debit note, freight invoice.

  • O2C (order-to-cash). Document families: sales order, sales invoice,

advance shipping notice (ASN), customer credit memo, dunning notice, statement of account, remittance advice, dispute correspondence.

2.2 The seven primitives

PrimitiveOne-line jobCanonical metric family
parseTurn raw bytes → structured fields + line items + table cells.field-P/R/F1, cell-P/R, TEDS, line-item count accuracy.
classifyAssign each input doc to a doc-type label.macro-F1, AUROC, confusion-matrix entropy, per-class P/R.
splitCut a bundle (multi-doc PDF) into logical sub-documents at the correct page boundaries.boundary-F1, page-attribution accuracy, full-bundle exact match.
matchTie one doc to others (3-way / 2-way / N-way) by IDs, line items, totals, dates, parties.per-leg precision/recall, end-to-end match rate, dollar coverage.
validateCheck extracted/matched data against business rules (totals, tax, duplicate, date sanity, party allow-list).rule-coverage F1, false-block rate, calibration vs human override.
augmentEnrich extracted data via lookups (master data, tax registries, PO-line catalogue, FX rates, holiday calendars).enrichment hit-rate, lookup precision, freshness, FX-error rate.
reasonDecide exceptions: "approve", "hold", "route-to", "ask for", with a justification trace.decision agreement vs human, hallucination rate, traceability score.

3. Reusability convention — how cells inherit from the framework

3.1 Inheritance rules

  1. Structure is fixed. Every cell-plan uses the di-study's nine-section

spine in §0.1 above. Sub-section numbering (3.1–3.8, 4.1–4.3, etc.) is preserved.

  1. Methodology is fixed. §3.1–3.8 carry the same wording skeleton; only

per-primitive specifics change.

  1. Scoring-rubric legend is fixed. Cells do not invent new statuses;

they use the eight-row legend in §0.4.

  1. Δ-rule is fixed. Non-overlapping 95% CIs only. No "we are slightly

better." Cells do not invent a winner.

  1. Numbers are placeholders. Every result cell reads [PLACEHOLDER] or

[PENDING-MEASUREMENT]. Cells do not invent values; they specify what would be measured.

  1. Dimensions are subset, not invented. Cells choose from D1–D14;

per-primitive metric specialisations (e.g. boundary-F1 for split) are declared as instances of the parent dimension (D2 in that case), not as new D-ids.

  1. Crosswalk is preserved. Every cell's §6 maps its dimensions onto

the 200/204-item taxonomies and explicitly names framework-introduced dimensions as study-introduced gaps.

  1. Capability coverage is HyperAPI-sourced, not measured. §7 in each

cell references the relevant HyperAPI SDK endpoint(s) (e.g. parse, parse-x, parse-intent, split, classify, extract, extract-omni, redact, detect, validate, reconcile, bundle) and notes Fully / Partially / Not yet from the HyperAPI capability inventory. No measurement language in §7.

  1. Competitor section is honest. §8 names the competitor SKU/version

only where a marketing source URL is recorded. No claim about a competitor without a citation.

  1. Findings section is gated. §9 is [PLACEHOLDER] until the cell

has at least one measured Δ with non-overlapping CIs. No "directional" findings.

3.2 What changes per cell

  • Document set (P2P doc types vs O2C doc types).
  • Per-primitive metric specialisation (e.g. classify uses macro-F1 +

per-class P/R; split uses boundary-F1; match uses dollar-coverage; etc.).

  • Sample-design strata (e.g. for classify, strata are doc-type frequency

bands; for split, strata are bundle-page-count bands).

  • Failure-mode matrix rows (e.g. for match, rows are

exact-key / fuzzy-key / partial-quantity / FX-cross / multi-leg combinations).

  • Limitations (each primitive has its own honest limit, e.g. reason has

fuzzy ground truth).

3.3 What never changes

  • The nine-section spine.
  • The data-status legend (eight rows).
  • The Δ-rule (non-overlapping 95% CIs).
  • The honesty discipline (no faked cells, every cell rerunnable or

PENDING).

  • The independence/affiliation disclosure.
  • The IDP-pipeline-position figure (parse → split → classify → extract

→ augment → validate → match → GL → reason), with the in-scope stage highlighted in green.
