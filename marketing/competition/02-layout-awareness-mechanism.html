Why layout awareness drives extraction accuracy — the mechanism

A document is a 2-D object. Text on a page carries meaning *because of where it

sits* — the number to the right of "Total Due" is the total because of its

position, not because of the digits themselves. A document-AI system that treats

a page as a 1-D stream of tokens has thrown away the signal that disambiguates

fields. This post is about the mechanism: *why* layout awareness moves the

accuracy needle, not a claim about how much.

The failure layout awareness prevents

Recall the error-mode taxonomy. Three of its seven buckets are *purely* layout

failures:

  • layout_misread — characters right, assigned to the wrong field. The total

pulled from the subtotal line. Most common on multi-column statements.

  • reading_order_error — text fragmented across the page and concatenated in

the wrong order. Most common with column-spanning tables.

  • table_structure_error — row/column boundaries drawn wrong; one logical row

split or two merged.

None of these is a character-recognition problem. A perfect OCR engine with no

layout model commits all three. That is the core point: **OCR accuracy and

extraction accuracy are different things**, and the gap between them is layout.

The mechanism, step by step

Consider an invoice with a two-column header — bill-to on the left, ship-to on

the right — and a line-item table below.

1. A reading-order-naive system linearizes the page top-to-bottom,

left-to-right. The two header columns interleave: "Acme Corp / Globex Inc /

123 Main St / 456 Oak Ave …". The bill-to and ship-to addresses are now

shuffled together. The characters are all correct. The fields are all wrong.

2. A layout-aware system first segments the page into regions — left column,

right column, table — and *then* reads within each region. The bill-to block

stays intact. Same OCR, correct fields.

The difference is not in the recognition model. It is in whether the system has a

representation of the page's 2-D structure before it commits to field

assignment. This is why the benchmark scores layout *separately* from

transcription.

Why we measure it separately — PaIRS

SAVIOR-Bench scores two things on identical inputs:

  • Transcription — word-level multiset recall and precision. Did the right

words come out, regardless of arrangement.

  • Layout fidelity — PaIRS, a pairwise spatial-relationship score. Did

the system preserve the relative spatial structure of the page.

A system can score high on transcription and low on PaIRS — it read every word

and arranged them wrong. That is precisely the layout_misread /

reading_order_error failure made visible. If you only measured transcription,

you would rank that system as excellent and then watch it mis-post totals in

production.

An honest caveat, stated up front: in SAVIOR-Bench, PaIRS is directly

comparable only between *prompt-controlled* systems. Across non-prompt-controlled

systems it is a format-compatibility proxy, not a strict ranking. We will not

present a clean PaIRS leaderboard across mixed systems and call it a ranking; it

is direction-only.

What the Hyperbots data does and does not show

The internal comparative study is candid here, and we keep that candour:

  • On SAVIOR-Bench, **fine-tuning a small open backbone did not buy layout

fidelity.** Strong zero-shot systems led on PaIRS by a wide margin

(category-label figures: 0.8766 vs 0.7066 for the fine-tuned row). Fine-tuning

bought recall and word-error-rate, not layout.

  • That is a finding *against* a naive "fine-tune wins everything" story, and we

report it because cropping it would misrepresent the result.

So the honest claim is not "HyperAPI has the best layout model." The honest claim

is: **layout is a distinct, measurable axis; most single-number accuracy claims

do not measure it; and any serious evaluation must.** Where HyperAPI stands on

that axis against a named competitor like the LLM-document-parsing API is PENDING MEASUREMENT — see

post 04 and DATA-NEEDED.md.

What this means for evaluation

When a vendor says "layout-aware," ask the mechanism question:

  • Does it segment regions before assigning fields, or linearize first?
  • Does it score layout *separately* from transcription, or fold both into one

accuracy number?

  • Can it show you a reading_order_error rate on multi-column documents?

A vendor that cannot separate "read the words" from "put the words in the right

field" has not measured the thing that actually breaks in production.
