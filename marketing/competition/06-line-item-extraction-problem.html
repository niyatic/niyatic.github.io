The line-item extraction problem — where most document-AI APIs break

Header fields are the easy part of an invoice. Vendor name, invoice number,

total — one value each, usually in a predictable place. The line-item table is

where document-AI systems actually fail. This post is about *why*, and how to

measure it honestly.

Why tables are hard

A line-item table is a 2-D structure with semantics:

  • It has a variable number of rows — 1 row or 200, unknown in advance.
  • Each row binds several cells together: qty, description, unit price, line

total are one logical record.

  • Rows can wrap — a long description spills onto a second visual line that is

still the same logical row.

  • Columns can be implicit — separated by whitespace alignment, not ruled

lines.

  • Tables span pages — a row order that breaks across a page boundary.
  • Subtotals, tax lines, and section headers sit *inside* the table region but are

not line items.

Every one of these is a place to go wrong. The error-mode taxonomy has a

dedicated bucket — table_structure_error: row/column boundaries drawn wrong,

one logical row split into two, or two merged into one.

The four ways line-item extraction breaks

1. Row splitting. A wrapped description is read as two rows. Now you have a

phantom line item with a description fragment and no price. `table_structure_

error`.

2. Row merging. Two tightly-spaced rows are collapsed into one. The qty of

row 1 binds to the price of row 2. The arithmetic no longer checks.

table_structure_error.

3. Column drift. On a whitespace-aligned table, the unit-price column and the

line-total column are adjacent and similar in magnitude. The model assigns the

line total to the unit-price field. Characters correct, field wrong — a

layout_misread.

4. Non-item contamination. A "Subtotal" line inside the table region is

extracted as a line item. Now your line-item sum double-counts.

A header-field accuracy number tells you *nothing* about any of these.

How to measure it honestly: row-level F1

The wrong metric is cell-level accuracy — it gives partial credit to a broken

row. The right metric is row-level F1:

  • A predicted row counts as a true positive only if all its cells are correct

and they are bound to the correct logical row.

  • Recall = correct rows ÷ gold rows. Catches row-merging and dropped rows.
  • Precision = correct rows ÷ predicted rows. Catches row-splitting and

non-item contamination (phantom rows).

  • Row-F1 = the harmonic mean.

Report it stratified — a 3-row table and a 90-row table are different difficulty

classes — and report n and a 95% CI. A useful secondary check is an

arithmetic-consistency rate: on what fraction of invoices does

sum(line_totals) reconcile to the document subtotal? That is a free,

gold-free signal that catches row-merge and column-drift errors.

parse-intent F1/WER. It does not currently publish an isolated **row-level F1

column** for line items. So we cannot put a HyperAPI line-item row-F1 number on

Totl ≠ Total. And the parse-intent WER result (0.2589 for the fine-tuned

signal that *correlates with* but is not the same as table fidelity.

Any "HyperAPI extracts line items more accurately than the LLM-document-parsing API" claim is

PENDING MEASUREMENT — it requires a shared dataset with gold line-item tables,

DATA-NEEDED.md.

What to demand of a vendor on line items

1. A row-level F1 number, not a cell-level or blended one.

2. Row-F1 stratified by table size and by document quality.

3. An arithmetic-consistency / reconciliation rate on real invoices.

4. The table_structure_error rate from an error-mode breakdown.

5. Behaviour on multi-page tables and wrapped descriptions specifically —

ask for example documents.

If a vendor's line-item evidence is a blended page-level accuracy number, they

have not measured the part of the document that actually breaks.
