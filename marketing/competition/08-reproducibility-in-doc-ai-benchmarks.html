Reproducibility in document-AI benchmarks — why most comparisons can't be trusted

A benchmark number you cannot rerun is a marketing asset, not a measurement. Most

published document-AI comparisons — including, until proven otherwise, ours —

cannot be reproduced by a stranger. This post explains the discipline that makes

a number trustworthy, and is candid about where the Hyperbots numbers currently

sit on that ladder.

The benchmark-factory bar: a number that does not "exist"

The factory constitution states it bluntly: **"Numbers without a rerunnable

command do not exist."** Concretely, a trustworthy cell requires:

  • Model pinning — exact model ID *and* revision/commit hash. No floating

latest. An API model pinned to its version date; a local model to a file

sha256.

  • Dataset pinning — name + version/snapshot + split + a content sha256

manifest. Splits frozen at registration so nobody can re-slice after seeing

results.

  • Harness pinning — the git commit of the harness, plus a PINS.txt with the

Python version, every dependency at an exact version, and the seed.

  • Decoding pinning — temperature, top-p, max-tokens, stop sequences fixed

*per task family*, identical across every model in that family.

  • A literal command stored with the cell — re-running it on the pinned

commit reproduces the metric within Monte-Carlo CI.

If any of these is missing, the comparison is not reproducible, and a 0.4-point

gap between two vendors is indistinguishable from noise.

Why most public comparisons fail this bar

  • Floating models. A vendor benchmarks against "GPT-4o" with no version date.

The model changes under them; the number is stale and unrepeatable.

  • Self-reported numbers. Vendor A benchmarks Vendor B by reading Vendor B's

marketing page. Neither ran the other under a common harness. The anti-gaming

rule: *submitters provide an adapter + a pinned model ref; the factory runs it,

the submitter does not self-report.*

  • Per-model prompt engineering. Vendor A tunes the prompt for its own model

and runs the competitor on a generic prompt. The factory rule: one frozen

prompt template per task family, public, identical for every model.

  • No contamination check. If the eval documents were in a model's

pretraining data, the score is memorization, not extraction. The factory

requires an n-gram/canary overlap audit per (model, dataset) and prefers

held-out data released after the model's training cutoff.

  • No CIs. Without a 95% CI you cannot tell a real win from sampling noise.

Overlapping CIs are a *tie band*, not a ranking.

A comparison that fails several of these is not "roughly right" — it is

unfalsifiable.

The honest part: where the Hyperbots numbers sit

We hold ourselves to this bar in public, so here is the unflattering truth.

SAVIOR-Bench v1's numbers are attested, not factory-reproduced. On

2026-05-21, the W6 workstream ran the pinned factory harness against all 23 rows

of the parse-intent table. The outcome: 0/23 cells re-derived green. Three

structural reasons, all logged:

1. The SEED harness is, by design, offline / stdlib-only / no-API-key — it has

no adapters for the 23 hosted-API vendor models.

2. The in-house parse-intent corpus is not in the benchmark repository, so

its loader is not registered.

3. The attested source carries only aggregate F1/WER per model — no

per-document score vector exists, so a bootstrap CI cannot be derived without

fabricating per-document data, which the team refused to do.

The honest reading — and we publish it exactly this way — is: the harness did

not produce a *competing* measurement that disagrees with the attested one; it

produced *no* measurement. Absence of independent re-derivation is not evidence

of error, but it is also not reproduction. So every SAVIOR-Bench number is read

at the attested tier: *in-house-run, user-attested 2026-05-18, not

factory-harness-reproduced, ci95=null.*

What W6 *does* contribute is evidence of due diligence: the harness was actually

run, the blockers were named, and **nothing was silently flipped, retracted, or

re-derived to look better.** That is the difference between an honest benchmark

and a marketing benchmark — not that the honest one has no gaps, but that it

names them.

What this means for evaluating any comparison — Reducto's, ours, anyone's

When a vendor shows you a comparison, ask:

1. Are all models pinned to a revision/version date?

2. Was the competitor run under a common harness, or self-reported from a

marketing page?

3. Is the prompt template frozen and identical across models?

4. Is there a contamination audit, and is the data held-out?

5. Is there a n and a 95% CI on every cell?

6. Is there a literal rerunnable command?

7. If a number cannot be reproduced, does the vendor *say so* — or imply it is

more verified than it is?

Question 7 is the tell. A vendor that labels its evidence tier honestly —

"attested, not reproduced" — is more trustworthy than one with a glossier number

and no audit trail. The asset is the honesty, not the gloss.
