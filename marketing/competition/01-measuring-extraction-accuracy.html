How to actually measure document-AI extraction accuracy

Most document-AI vendor pages quote a single number — "99% accuracy" — with no

field breakdown, no document-quality stratification, no n, and no confidence

interval. That number is not measurement. It is a headline. This post is the

opposite: the metric framework HyperAPI (by Hyperbots) uses internally, written

out in full so you can apply it to *any* vendor, including us.

Why a single accuracy number is meaningless

"Accuracy" on a document is not one quantity. A document has a vendor name, an

invoice number, a total, a tax line, a date, a dozen line items, and a payment

terms note. Each is a different recognition problem. A model can nail the header

fields and still mangle every line-item row — and a blended "98%" hides that

completely. The blended number also hides *which documents*: a clean born-digital

PDF and a skewed mobile photo are not the same task.

The HyperBots benchmark-factory discipline states the rule plainly: **accuracy is

never reported as one number.** It is reported on a 2-D grid of field-type ×

document-quality, each cell carrying its own n and 95% confidence interval.

The metric framework, dimension by dimension

1. Field-level precision / recall / F1

For each extracted field, score it against gold:

  • Precision — of the fields the model emitted, what fraction were correct.

Low precision means the model invents or mis-fills fields (a

schema_mismatch_hallucination in the error taxonomy below).

  • Recall — of the fields that should have been extracted, what fraction the

model got. Low recall means the model drops fields (a refusal_no_output or a

missed region).

  • F1 — the harmonic mean. The honest single metric *per field type*, never

blended across all fields into one page-level number.

Precision and recall trade against each other, and which one matters depends on

your downstream process. An AP team that auto-posts invoices fears low precision

(wrong total posted) far more than low recall (a field flagged for review).

2. Layout-aware accuracy

Getting the characters right is not enough if they land in the wrong field. The

classic failure: the model reads 1,234.00 perfectly but pulls it from the

*subtotal* line instead of the *total* line. Character-level scoring calls that a

win. It is a layout_misread and it is a real production defect.

A layout-aware metric scores whether the extracted value is associated with the

correct semantic region. SAVIOR-Bench (the Hyperbots-side benchmark) uses

PaIRS, a pairwise spatial-relationship score, for exactly this — does the

model preserve the relative spatial structure of the page, not just the glyphs.

Note the honesty caveat in the benchmark: PaIRS is only directly comparable

between *prompt-controlled* systems; across non-prompt-controlled systems it is a

format-compatibility proxy, not a strict ranking. We state that rather than imply

a clean ranking.

3. Line-item / table fidelity

A table is not a bag of cells. A purchase-order table has rows, and a row binds

qty + description + unit-price + line-total together. The right metric is a

row-level F1: a row counts as correct only if its cells are correct *and*

bound to the same logical row. The error taxonomy bucket here is

table_structure_error — one logical row split into two, or two merged into one.

This is covered in depth in companion post 06.

4. Numeric and locale correctness

($125.00) is negative 125. 1.234,56 is one-thousand-two-hundred in a European

locale. A model that recognizes every digit but mishandles sign or locale commits

a numeric_format_error. For finance documents this is not cosmetic — it changes

the posted amount. Numeric fields get a numeric-tolerance metric, not a string

match.

5. Confidence intervals and n

Every cell carries a 95% confidence interval and the n it was computed on.

The interval is a nonparametric bootstrap (≥1000 resamples, pinned seed) over

per-item scores. Two consequences the factory enforces:

  • If two models' 95% CIs overlap, they are reported as a tie band, not

strictly ranked. "We beat them by 0.4 points" on overlapping CIs is not a real

win.

  • A cell with n < 100 is marked low-confidence and excluded from any aggregate.

An honesty note from the Hyperbots comparative study: SAVIOR-Bench v1 currently

ships with ci95=null — the attested source carries only aggregate F1/WER per

model, not per-document score vectors, so a bootstrap CI cannot be derived

without fabricating per-document data, which the team refused to do. The honest

move is to publish ci95=null and call every delta "direction-only," not to

invent an interval. That is the standard this whole framework is held to.

The error-mode taxonomy

A wrong extraction is not just "wrong" — it is wrong in one of seven specific

ways. Bucketing failures tells you what to fix:

bucketwhat happened
ocr_character_errorwrong character recognized (low-DPI, mobile capture)
layout_misreadright characters, wrong field
reading_order_errortext fragments concatenated in the wrong order
schema_mismatch_hallucinationa field invented that is not in the source
numeric_format_errorsign/locale mishandled
table_structure_errorrow/column boundaries drawn wrong
refusal_no_outputmodel declined or returned empty

A good evaluation reports the top three buckets per system, with an example

document each. A vendor that cannot tell you *how* it fails has not measured

itself.

What to demand of any vendor — including HyperAPI

1. Field-level P/R/F1, broken out by field type — not one blended number.

2. A document-quality stratification (clean digital / scanned / mobile /

multilingual / handwritten).

3. Row-level F1 for tables, not cell-level.

4. n and a 95% CI on every cell — or an honest ci95=null with the reason.

5. The error-mode breakdown.

6. A rerunnable command behind every number.

If a vendor — us or anyone else — cannot produce this, the accuracy claim is

unverified. Treat it accordingly.
