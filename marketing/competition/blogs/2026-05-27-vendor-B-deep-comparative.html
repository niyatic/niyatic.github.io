LLM-only parsing APIs vs primitive-resolved finance IDP — a stack-level honest comparison

A new band of vendor is taking up shelf space in the doc-AI category: the LLM-only document-parsing API. The pitch is clean. Send a PDF. Get structured output back. Tables, layout, text, ready for retrieval. One endpoint, one bill, one mental model. For a developer wiring a RAG pipeline next Tuesday, that shape is a real win.

For a finance team turning a folder of mixed PDFs into approved postings in a general ledger, that shape is one primitive of seven. This post is the honest map of where the LLM-only parsing band lands, where it stops, and what a finance doc-AI stack actually has to do.

We do not name vendors in this post. The named comparison — vendor name, endpoint inventory, pricing structure, public-claim audit — lives in an internal gated study at content/competitor-studies/vendor-B/STUDY-2026-05-27.md. This post uses category labels only. The two labels that matter here:

  • LLM-only document-parsing API — a developer-facing parsing API where the dominant call shape is a single "parse this document" call backed by an LLM or VLM, returning text + layout + tables + optional schema-driven extraction, shaped for downstream AI pipelines.
  • API-native, finance-domain, primitive-resolved IDP — an API that exposes parse, classify, split, match, validate, augment, reason as named, addressable endpoints, with a model fine-tuned for financial documents.

The question this post answers: what does each band do, and where does the comparison actually live?

1. Why "send a PDF, get JSON back" is a category, not just a product

The LLM-only document-parsing band did something useful for the category. It collapsed three things that used to need three vendors — OCR, layout analysis, table extraction — into one API call. It did this by riding the zero-shot capability of frontier vision-language models, which became good enough at general-purpose document reading that the classical OCR-plus-layout-plus-table pipeline could be replaced by a single model call.

For workloads where the document is the input to something else — a retrieval index, an agent loop, a downstream extraction prompt — that collapse is the right shape. The developer's job is "documents in, chunks out." A single-call API hits that job description exactly.

This is also where the category's center of gravity sits today. The fastest-growing slice of the LLM-only parsing band is the RAG- ingestion workload: feed scientific papers, contracts, technical reports, or long PDFs into a retrieval pipeline that an LLM agent reads. The general-purpose nature of the band is its advantage; the band's training is on a wide document distribution, and a wide document distribution is exactly what a developer's RAG corpus looks like.

2. What the LLM-only band does well — said honestly

Before the gap, the honest acknowledgement. A vendor disagreement that doesn't acknowledge the other side's strengths is a sales page, not a comparison.

  • Out-of-domain generality. A frontier-VLM-backed parsing API inherits zero-shot capability across document classes that a finance- domain-tuned model has never seen — research papers, legal exhibits, scientific PDFs, technical reports, long-form contracts. On the Hyperbots-published SAVIOR-Bench v1 panel, the zero-shot systems beat the fine-tuned ones on precision and layout dimensions. The fine-tune edge in our own published numbers is real but narrow — +0.0276 F1 on parse-intent, +0.0061 recall on OCR. The general-purpose models are not a punching bag.
  • Time-to-first-call for a developer. A parsing API shaped around a single, well-documented endpoint is the right onboarding shape for an engineer building a retrieval pipeline. The developer-loop is fast.
  • Embedding / chunking-ready output. A parsing API that produces retrieval-ready chunks, table-as-Markdown, and embedding-friendly text is doing useful work for the RAG band. That work is real and it is hard to do well.
  • Frontier-model evolutionary inheritance. A wrapper on a frontier VLM inherits the next generation of that VLM for free. The trajectory is real.

The honest finding behind every Hyperbots benchmark post is that the LLM-only parsing band is better than the market gives it credit for on what it actually does, and wrong for the surface it does not expose. Both halves of that sentence matter.

3. Where the LLM-only band stops

The seven-primitive map of a finance doc-AI stack — parse, classify, split, match, validate, augment, reason — names where the band stops. This is the load-bearing section.

3.1 parse — primary surface, real strength

This is where the LLM-only band lives. A single-call API turning a PDF into structured layout + tables + text is doing the parse primitive's job. Across most document classes, on most cases, the work is good. The known failure modes — line-item drift on long invoices, schema hallucination, spanning-header tables, the long-tail bilingual carrier document — show up on the hard cases, not the average case.

For a finance buyer, the honest read is that an LLM-only parsing API serves the parse primitive as a primitive. On the average invoice, the average bill of lading, the average remittance, the work is real.

3.2 classify — possible at the prompt layer, not a named endpoint

Classification — assigning each input document to its doc-type label (PO, GRN, invoice, BoL, packing slip, credit note, debit note, freight, remittance, statement, dispute) — is something a downstream prompt can do over the parsing API's output. It is not, in the LLM-only band, typically a named endpoint with a documented schema and a published metric. The Δ is not "can it classify" — it is "is classify a first- class object the buyer can address."

The right metric for classify is macro-F1 plus per-class P/R plus a confusion-matrix entropy view, not a single "classification accuracy." A buyer who wants to know how the classifier handles the remittance-vs-statement confusion pair, or the freight-vs-supplier- invoice confusion pair, needs the metric exposed.

3.3 split — structurally weak in single-call shape

Split — cutting a bundle PDF (fifty pages of mixed documents emailed in by a supplier) into the right sub-documents at the right page boundaries — is the primitive that gets the least attention and the one that silently bottlenecks production.

An LLM-only flow has no native sense of "document boundary"; it sees page tokens. Boundary-F1 and page-attribution accuracy are not metrics the band ships against. A buyer with bundle-PDF inputs is buying around this gap, not from it.

3.4 match — not a fit for the band's shape

Match — tying one document to others (3-way / 2-way / N-way) by IDs, line items, totals, dates, parties — is a database operation against a corpus of POs and GRNs. An LLM-only parsing API has nowhere to put that corpus. Three-way match is what finance teams in P2P are actually paying for; cash application (the O2C match cell) is what finance teams in O2C are actually paying for. Both require a corpus, a tolerance policy as configuration, an FX cross-rate handling, a reason-code agreement metric.

The honest sentence is three-way match is not a parsing problem, and a parsing API is not the right shape for it. A finance buyer who needs match is buying a different surface area.

3.5 validate — auditability gap

Validation against business rules — totals, tax, duplicate, date sanity, party allow-list — is doable in a prompt to an LLM. The auditability of that path is the gap. When the rule is "tax = rate × base, tolerated to 0.01 absolute or 1% relative, with currency normalisation pinned to today's FX table," and the rule lives as a string the model decides whether to honour, a finance controller cannot audit the rule catalogue as a first-class object.

A primitive-resolved API exposes validate with per-rule coverage and false-block rate as a first-class metric. That is what a controller's RFP asks for. The LLM-only band's path here is workable on the demo and fragile in audit.

3.6 augment — plumbing, not parsing

Augment — enrichment via lookups against master data, tax registries, PO-line catalogues, FX rates, holiday calendars — is plumbing. The right metric is enrichment hit-rate, lookup precision, freshness (how stale is the FX rate?), and FX-error rate. An LLM-only parsing API is not the master-data store; integration with the master-data store lives elsewhere. The Δ is not "the band can't do augment" — it is "the band is not the augment primitive's home, and the buyer pays for plumbing to land it."

3.7 reason — decisioning with a justification trace

reason is the hardest primitive. The output is not a single right answer; the ground truth is a distribution of human decisions. The right metrics are decision-agreement-vs-human, off-distribution decision rate (cases where the system picks an option no human picked), and justification-trace fidelity (does the trace cite the fields, rules, and lookups that actually drove the modal human decision?).

An LLM-only parsing API was never built for this. A decision-bearing endpoint with a citation-trace is a different product shape. The finance-domain primitive-resolved band ships this as a first-class primitive; the LLM-only parsing band does not.

4. The five real gaps a finance buyer should test

Independent of vendor name, five gaps separate the bands. Each is testable on the buyer's own documents.

4.1 Endpoint inventory — addressable primitives, not "one big call"

Ask the candidate vendor for the named endpoint list. Count how many of the seven primitives are first-class — i.e. have a documented schema, a published metric, and a price line. A band that exposes parse (sometimes with an extract adjacency) and waves at the others is selling one primitive. A band that exposes all seven is selling the stack.

This is the architectural Δ between the LLM-only band and the primitive-resolved finance band. It is structural, not a measurement question. It is observable in five minutes on the vendor's docs page.

4.2 Pricing axis — predictability vs document size

Per-page pricing is structurally predictable to a finance buyer. A hundred invoices today, fifty tomorrow, two hundred Monday — the bill is the page count times the per-page rate. Per-token pricing scales with document length and with prompt size. For a finance buyer forecasting a budget over a quarter, that is a different shape of risk.

The honest version of this section is: an LLM-only parsing API in practice often prices per-page or via a credit system, but the band's underlying cost is per-token. Where the band's pricing exposes that axis to the buyer, the buyer is taking the token-cost risk. Where it doesn't, the buyer is taking the vendor's risk and paying the spread.

For reference, the published per-page price on the finance-domain primitive-resolved band's side is $0.05/page — disclosed inline because it is the published anchor. Volume is sized on a call. We do not assert a Δ against the LLM-only band's pricing here, because the named vendor's pricing axis is verified inside the gated study, not in public copy.

4.3 Domain depth — finance-tuned vs general-purpose

A model fine-tuned on financial-document variance — supplier invoices with spanning-header tables, multi-leg bills of lading, remittance advices with line-by-line short-pays, bilingual carrier docs — is a different unit of work than a general-purpose VLM. Per the SAVIOR- Bench v1 panel, the fine-tune edge is +0.0276 F1 on parse-intent, +0.0061 recall on OCR vs the best zero-shot system. Real but narrow.

The honest read of "real but narrow" is twofold:

  1. Don't expect the finance-tuned band to blow out the general- purpose band at the average-case parse on a financial document. The edge is small.
  2. Do expect the gap to live on the long tail — the spanning-header table, the multi-leg BoL, the long supplier- invoice with sub-totals, the photographed delivery docket. That is where a finance-tuned backbone earns the bill.

A buyer should run their hardest 50 documents through both bands and measure F1 on those, not on the easy cases the demo covers.

4.4 Architecture — single engine vs multi-engine, mode-routed

An LLM-only parsing API is, by construction, a single-engine stack: the LLM is the engine. A primitive-resolved finance API can compose several engines under per-document mode routing — invoice OCR, receipt parsing, bank-statement extraction, and shipping-document parsing are different problems that the same engine plays differently on. The composition pattern is itself the architecture.

A finance buyer's production load is not one document class; it is a mix. A mode-routed multi-engine stack is shaped for that mix. A single-engine LLM stack is shaped for a single distribution.

4.5 Benchmark and audit discipline — reproducible, with a published failure

The last gap is methodological. Does the candidate vendor publish a reproducible benchmark — pinned dataset name + version + n, rerunnable command, point estimate + 95% CI, distinction between Missed / Spurious / Incorrect, an affiliation disclosure?

For reference: the finance-domain primitive-resolved band's published anchors include the SAVIOR-Bench v1 panel (509 docs, IAA 0.761), a SEED leaderboard (12 cells, designed to be reproducible on a laptop with stdlib-only deterministic primitives), and a W6 audit that re-derived 0 of 23 SAVIOR cells under the factory-reproduction harness — published as the audit's finding, because the negative finding is the asset. Whether a candidate LLM-only parsing API publishes a comparably structured benchmark is itself a buyer-rubric question, and the answer for any specific vendor lives in the gated study.

6. What to do with this — for the buyer, for the practitioner

For a finance buyer evaluating both bands: the question is not "which band has higher parse accuracy?" — the difference is too narrow for that to be the decider. The question is which seven- primitive surface do you actually need? If your job is to feed documents into an LLM agent for downstream prompting, the LLM-only parsing band is built for you. If your job is to turn a folder of mixed PDFs into approved postings in a general ledger, with an audit trail and a three-way match, the LLM-only band serves one of seven primitives and you are buying around the other six.

For an ML or platform practitioner: pick one of the seven primitives and run both bands against the same documents under identical scoring. The deep-study framework at content/deep-study-framework/FRAMEWORK-2026-05-25.md gives the fourteen-cell plan; pick a cell, take its plan as a template, run it. The asset is the methodology, not the result; the result is what publishes the cell.

For a CIO or finance-tech executive: ask the candidate vendors to show their per-primitive resolution. A vendor that can only show "parsing accuracy" is selling one primitive and waving at six. That is not a stack — that is a feature.

A vendor that can show seven addressable endpoints, with metrics per primitive, a pinned dataset, a rerunnable command, and a published audit that includes its own failures — that is a stack.
