Why we built HyperParse

General document parsers are tuned on broad web-scale corpora — research papers, news articles, contracts. Finance documents are a different beast. An invoice from a Japanese vendor may carry a JPY-to-USD conversion table in a rotated side column. A bank statement for a multi-currency account may repeat a "balance" column three times across a three-page spread, with each instance representing a different currency. A cheque image may arrive as a PNG scan with MICR (Magnetic Ink Character Recognition) encoding in a proprietary font at the bottom of the page — most general VLM-based parsers hallucinate or drop MICR lines entirely because the character set is outside their training distribution.

We saw this pattern repeatedly: high-quality general parsers that handle a legal contract cleanly would fail on a remittance advice with a 40-row payment reference table, because the table header appears on page 1 and the rows run across pages 2–4, with no visual separator between the continuation rows and the following total-line. The failure mode is not OCR — the characters are read correctly. The failure is structural: the parser can't infer that the orphaned rows on page 2 belong to the same table as the header on page 1. HyperParse was built to get this right.

What HyperParse is

HyperParse is the document-understanding layer inside the HyperAPI SDK. It exposes four callable primitives:

from hyperapi import parse, classify, split, extract

# Parse a document to structured markdown / layout-aware output
result = parse(file_path="invoice.pdf")

# Classify document type
doc_type = classify(file_path="unknown.pdf")

# Split a multi-document PDF into individual document boundaries
pages = split(file_path="batch.pdf")

# Extract named fields from a parsed document
fields = extract(file_path="remittance.pdf", schema=my_schema)

Two additional capabilities are on the roadmap:

  • parse-intent — semantic extraction with intent-aware field resolution. The hyperapi-parse-intent model has a factory-attested benchmark cell (F1 0.8328, see below), but the SDK primitive is not yet productized. Status: PENDING-SDK.
  • bounding-box / visual grounding — field-to-region localization. The benchmark task facet is defined; no measured cell exists yet. Status: PENDING-DATA.

Product accuracy claims are published at apis.hyperbots.com and are not reproduced in this benchmark post. The metrics below are model-level research numbers from our internal benchmark factory.

HyperParse is tuned for the five document types that drive the highest friction in accounts-payable and accounts-receivable automation:

  • Invoices — line items, tax structures, continuation pages, ERP-schema mismatches
  • Bank statements — multi-page tables, running balances, transaction classification
  • Cheques — mixed PDF and PNG source formats, MICR field extraction, handwritten amounts
  • Credit notes — visually similar to invoices but with polarity reversal on amounts
  • Remittance advice — payment references, multi-currency, payer/payee disambiguation

The same parse primitive handles all five. Routing to specialist sub-pipelines (OCR, VLM, structured extractor) is automatic.

What we DON'T claim

Honesty is a design requirement, not a disclaimer. Here is what this post does not assert:

  • No layout IoU benchmark. We have not run a bounding-box / region-detection benchmark. Extend's RealDocBench measures layout accuracy via Adjusted F1 with IoU thresholds against 1,500 human-annotated samples. We have no equivalent measurement on disk. Comparing our parse success rate to their layout Adjusted F1 would be apples-to-oranges; we don't make that comparison.
  • No head-to-head vs Extend Parse 2.0, Reducto, LlamaParse, or AWS Textract. We don't have API keys for these four providers. We cannot run them through our harness. Any row labeled PENDING-KEY in our leaderboard is an honest null, not a suppressed number.
  • No productized parse-intent or extract-omni numbers. The hyperapi-parse-intent model has a research cell (F1 0.8328, fine-tuned on SAVIOR-Train, in-house-attested). The SDK primitive is not yet callable. extract-omni is PENDING-SDK entirely.
  • parse-intent F1 is fine-tuned, not zero-shot. The 0.8328 figure is from a model fine-tuned on HyperBots IP (SAVIOR-Train, LoRA). It is not directly comparable to zero-shot baselines on the same leaderboard. The fine-tuned advantage over the best documented zero-shot row (GPT-4o at 0.8052) is +0.0276 F1.
  • Benchmark numbers are research metrics, not product accuracy. Per apis.hyperbots.com verbatim for product-level claims.

How to call HyperParse

All primitives are available via the HyperAPI SDK. For current API documentation, rate limits, and product-level accuracy specifications, see apis.hyperbots.com.

import hyperapi

# Parse — returns structured output (markdown + layout metadata)
parsed = hyperapi.parse(file_path="bank_statement.pdf")

# Classify — returns document type label
label = hyperapi.classify(file_path="unknown_doc.pdf")

# Split — returns page-boundary annotations for batched PDFs
boundaries = hyperapi.split(file_path="multi_doc_batch.pdf")

# Extract — returns named fields per your schema
extracted = hyperapi.extract(
    file_path="remittance.pdf",
    schema={"fields": ["payer_name", "total_payment_amount", "currency"]}
)

SDK installation, authentication, and versioning: apis.hyperbots.com.

  • The multi-corpus parse sweep (DC-006) used preproduction SDK 0fc4bada; production behavior may differ.

  • The parse-intent and OCR edit-distance cells (DC-000) are in-house-attested or externally-sourced; neither is factory-harness-reproduced.
