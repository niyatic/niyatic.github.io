Match (P2P) — deep-study plan

1. Executive summary (TEMPLATE — numbers placeholder)

The signature P2P operation: 3-way match = tie supplier_invoice ↔ purchase_order ↔ goods_receipt_note, plus 2-way (invoice↔PO when no goods are received) and N-way (invoice ↔ multiple GRNs / blanket POs). Tolerance policy (qty / price / total / date) is part of the methodology, not a free parameter. Numbers [PENDING-MEASUREMENT].

2. Scope & coverage

  • Workload. Triples (PO, GRN, invoice) drawn from realistic AP

workflows; subset of doubles (PO, invoice) and N-way bundles.

  • Platforms. Hyperbots match (or reconcile per HyperAPI scope) +

≥1 competitor.

  • Coverage dims. 8 D-dimensions; layout/script/scan dimensions

dropped (parse and classify already handled those upstream).

  • Test surface.
  • n = [PLACEHOLDER] triples per dim.
  • Stratification axes: match-type (2-way / 3-way / N-way),

per-leg quantity-/price-/total-tolerance band, currency-cross (cross-FX rate applied or not), partial-fulfilment (GRN < PO), over-ship (GRN > PO), invoice-includes-freight-from-separate-PO, blanket-PO-over-time.

  • Gold: AP-team-validated matches with reason-code for tolerance

excursions.

3. Methodology

3.1 Data-status representation

Eight-row legend, with per-leg evaluation: each match decision (PO↔invoice, GRN↔invoice, line↔line) is its own scoring unit.

3.2 Sample design

n = [PLACEHOLDER] triples per dim. Two-annotator + adjudicator per match (because AP humans themselves disagree on edge cases — that disagreement is itself a measured quantity). κ reported.

3.3 Metrics

  • Per-leg precision / recall / F1. Three legs (PO↔inv, GRN↔inv,

PO↔GRN line-by-line).

  • End-to-end 3-way match rate = fraction of triples whose full

match (all three legs) is Correct.

  • Dollar coverage = sum(invoice_total of correctly-matched

triples) ÷ sum(all invoice_total). The business-relevant metric.

  • Line-level match P/R/F1 after deterministic line alignment

(description Jaccard + qty/price tolerance).

  • False-block rate = fraction of triples humans would approve that

the system rejects (a precision-of-rejection metric).

  • Reason-code agreement = where the system holds for a tolerance

excursion, does it cite the same reason-code as the human?

  • 95% bootstrap CI + n.

3.4 Scoring rubric

  • Quantity-tolerance: max([PLACEHOLDER]% rel, [PLACEHOLDER] abs),

pinned.

  • Price-tolerance: same.
  • Total-tolerance: same; cross-currency comparisons use a pinned FX

table (snapshot date in PINS.txt).

  • Date-tolerance: [PLACEHOLDER] days, pinned.
  • A correct match with wrong reason-code is half-credit on the reason-

code agreement metric only; the underlying match itself is Correct.

3.5 Evaluation controls

Frozen sha256. Rerunnable command:


run --study match-p2p-v0 \
    --dimension D<id> \
    --system hyperbots-idp@<model-revision> \
    --system <competitor>@<api-version> \
    --dataset p2p-match-bench-v1@<snapshot-sha256> \
    --split test --seed <pinned> \
    --metric e2e_3way_match_rate --bootstrap 1000

3.6 Independence and disclosure

Hyperbots-published; affiliation declared. Special note: the tolerance policy applied to a competitor must mirror the policy applied to Hyperbots — same pinned thresholds, same FX snapshot. Otherwise the comparison is not apples-to-apples and is labelled as such per the benchmark-factory rule.

3.7 Limitations

  • Partial-fulfilment edge cases are policy-sensitive; pinned policy in

PINS.txt.

  • Blanket-PO-over-time strata require time-series ground truth which is

partner-restricted; flagged.

  • Reason-code taxonomy is customer-specific; a canonical 10-code set is

pinned for the public-bench tier.

3.8 Where this primitive sits in the IDP pipeline

The match stage, downstream of parse / classify / extract / augment / validate. Output feeds the GL-post stage and the reason stage (for exceptions).
