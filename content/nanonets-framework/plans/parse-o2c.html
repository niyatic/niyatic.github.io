Parse (O2C) — deep-study plan

1. Executive summary (TEMPLATE — numbers placeholder)

How well does an IDP system parse O2C document bytes — sales order, sales invoice, advance shipping notice (ASN), customer credit memo, dunning notice, statement of account, remittance advice, dispute correspondence — into the canonical parse schema? Distinct from the P2P side: O2C documents carry inbound payment artefacts, with cash-application semantics (open invoice → applied amount → short-pay → reason code) and statement-level totals/aging buckets rather than PO/GRN matching. 3. Methodology

3.1 Data-status representation

Eight-row legend per framework §0.4.

3.2 Sample design

  • n = [PLACEHOLDER] per dimension; stratified per §5.
  • Macro-averaged across strata; remittance advice line-item nuances

(open-invoice references, deduction codes) are over-represented in D8 strata.

  • κ per stratum.

3.3 Metrics

  • Field-level P/R/F1 for the O2C-schema header (invoice no., sales-order

ref, customer block, totals, aging-bucket headers).

  • Open-invoice-application line-level P/R/F1 for remittance:

open_invoice_id, applied_amount, deduction_code, deduction_amount.

  • Statement aging-bucket extraction P/R/F1 (current / 30 / 60 / 90 / 120+).
  • Table-structure fidelity: cell-P/R + TEDS.
  • Bounding-box quality (IoU distribution).
  • Point + 95% bootstrap CI + n; overlapping CIs → tie-band.

3.4 Scoring rubric

  • Exact-match (casefold + whitespace-collapsed) for identifiers.
  • Numeric-tolerance max(1% rel, 0.01 abs) for monetary fields.
  • Deduction-code normalisation: mapped through a pinned dictionary

(e.g. D-001 → short-shipment) before exact-match scoring.

  • Wrong value = FP AND FN.
  • Table structure scored independently of cell content.

3.5 Evaluation controls

  • Frozen snapshot sha256.
  • Rerunnable command:

`` run --study parse-o2c-v0 \ --dimension D<id> \ --system hyperbots-idp@<model-revision> \ --system <competitor>@<api-version> \ --dataset o2c-parse-bench-v1@<snapshot-sha256> \ --split test --seed <pinned> \ --metric field_f1 --bootstrap 1000 ``

  • Contamination review per document.

3.6 Independence and disclosure

Hyperbots-published; affiliation declared; GA competitor offerings.

3.7 Limitations

  • Remittance advice formats vary wildly across customer ERPs;

representativeness depends on §5 stratification.

  • Statement-of-account aging-bucket headers are often customer-specific

(>120 vs 120+) — pinned dictionary tolerates synonyms.

  • Dispute correspondence is free-form prose; parse F1 will be lower-

bounded by GT ambiguity, not by extractor quality.

3.8 Where this primitive sits in the IDP pipeline

parse stage only. Cash-application (match) and exception-routing (reason) are scoped separately.
