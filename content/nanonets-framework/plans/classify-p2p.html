Classify (P2P) — deep-study plan

1. Executive summary (TEMPLATE — numbers placeholder)

How well does an IDP system assign one of the eight P2P doc-type labels — PO / GRN / supplier_invoice / BoL / packing_slip / credit_note / debit_note / freight_invoice — to an inbound document? Distinct from parse (no field extraction) and from split (assumes already boundary-cut). The primitive of interest is the classifier head: input = one document; output = (label, confidence) and optionally a (class-conditional confidence vector). Numbers [PENDING-MEASUREMENT].

2. Scope & coverage

  • Workload in scope. Eight P2P labels (above) + an other reject class.
  • Platforms assessed. Hyperbots classify + at least one competitor.
  • Coverage dimensions. 11 D-dimensions (D4 and D11 dropped — currency

and bounding-box are orthogonal to a class decision; flagged in §3.7).

  • Test surface.
  • n = [PLACEHOLDER] per dimension, stratified by label frequency

band (head / torso / tail) so rare labels (debit_note, freight_invoice) are not drowned by supplier_invoice volume.

  • Frozen splits; per-class κ on label annotation.

3. Methodology

3.1 Data-status representation

Eight-row legend per framework §0.4. Per-class: Correct = top-1 label matches GT; Incorrect = top-1 differs; Missed = abstained when GT label exists; Spurious = predicted a label when GT is other.

3.2 Sample design

  • n = [PLACEHOLDER] per dimension. Label-frequency-band stratification.
  • Two independent label-annotators; κ per class; third-reviewer

adjudication on disagreements.

  • Macro-averaged over classes (not micro) so rare labels are not

outweighed.

3.3 Metrics

  • Per-class precision / recall / F1. Macro-F1 across the eight + 1

reject. Micro-F1 also reported but de-emphasised.

  • Confusion matrix. Row-normalised, with diagonal mass per class.

Entropy of the off-diagonal reported as a "structured-confusion" score — high values mean confusion is concentrated in one wrong neighbour (a fixable failure mode), low means diffuse.

  • AUROC per class (one-vs-rest) where confidence is exposed.
  • Reject-rate calibration: precision-at-coverage curve; what is the

precision when the classifier abstains on its lowest-confidence X%?

  • Point + 95% bootstrap CI + n.

3.4 Scoring rubric

  • Top-1 exact-label match is the primary unit.
  • Reject (other) is a first-class label, not silence; it is scored.
  • Tied top-2 confidence within [PLACEHOLDER — pin ε] → classifier

must declare abstention; declaring a top-1 anyway is scored as a D9 calibration failure.

3.5 Evaluation controls

  • Frozen sha256 dataset snapshot.
  • Rerunnable command:

`` run --study classify-p2p-v0 \ --dimension D<id> \ --system hyperbots-idp@<model-revision> \ --system <competitor>@<api-version> \ --dataset p2p-classify-bench-v1@<snapshot-sha256> \ --split test --seed <pinned> \ --metric macro_f1 --bootstrap 1000 ``

  • Contamination audit per document.

3.6 Independence and disclosure

Hyperbots-published; affiliation declared; GA offerings.

3.7 Limitations

  • Currency (D4) and bounding boxes (D11) are dropped — they do not bear on

a class label.

  • Tail labels (debit_note, freight_invoice) will have wider per-class CIs

even at n=5,000 if their natural prevalence is <2%.

  • "Other" / reject-class boundary is policy-dependent; pinned in PINS.txt.

3.8 Where this primitive sits in the IDP pipeline

The classify stage of parse → split → classify → extract → augment → validate → match → GL → reason. Upstream: parse/split. Downstream: extract decides which schema to apply based on this label.
