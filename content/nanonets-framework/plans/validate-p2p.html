Validate (P2P) — deep-study plan

1. Executive summary (TEMPLATE — numbers placeholder)

Given a matched / extracted P2P document set, validate applies business rules: arithmetic reconciliation (line × qty × price = line total; Σ lines = subtotal; subtotal + tax + freight = total), tax-rate correctness, duplicate-invoice detection, vendor allow-list / sanctions check, date sanity (invoice date ≤ due date), PO open-balance check, GRN existence for goods, and customer-policy rules (spend-limit / category-exclusions). The output is a verdict (pass/hold/reject) + rule-citation trace. Numbers [PENDING-MEASUREMENT].

2. Scope & coverage

  • Workload. P2P documents post-extract: supplier invoices + their

matched PO/GRN context.

  • Platforms. Hyperbots validate + ≥1 competitor.
  • Coverage dims. 7 of 14.
  • Test surface.
  • n = [PLACEHOLDER] invoices per dim.
  • Stratification: rule-family (arithmetic / tax / duplicate / sanctions

/ date / policy), excursion severity (clean / minor / material), invoice complexity tier.

  • Gold: AP-rule-book validated verdicts with rule-citation trace.

3. Methodology

3.1 Data-status representation

Eight-row legend. Per-rule scoring: each rule fires Correct / Incorrect / Missed / Spurious independently.

3.2 Sample design

n = [PLACEHOLDER] per dim. Two annotators per invoice; adjudicator; κ reported. Rule-coverage stratification: every rule family receives a minimum stratum quota so rare rules (sanctions hits) are not drowned out.

3.3 Metrics

  • Per-rule P/R/F1. Each rule has its own confusion matrix.
  • Rule-coverage F1 = how completely the system applies the rule

set when the rule applies.

  • False-block rate = invoices the system holds that humans

release.

  • Verdict agreement = top-level pass/hold/reject agreement vs

human, weighted by dollar.

  • Rule-citation accuracy = when the system holds, does it cite

the right rule(s)?

  • Calibration vs human override (D9) = if the system says "hold

with confidence c", how often is hold the right answer at confidence band c?

  • 95% bootstrap CI + n.

3.4 Scoring rubric

  • Arithmetic tolerance: max([PLACEHOLDER]% rel, [PLACEHOLDER] abs)

pinned.

  • Tax-rate tolerance: pinned (jurisdiction-specific rate tables in

PINS.txt with snapshot date).

  • Duplicate detection: pinned similarity threshold (key tuple +

fuzzy-amount).

  • A correct hold with the wrong rule cited is half-credit on rule-

citation; verdict is still Correct.

3.5 Evaluation controls

Frozen sha256. Rerunnable command:


run --study validate-p2p-v0 \
    --dimension D<id> \
    --system hyperbots-idp@<model-revision> \
    --system <competitor>@<api-version> \
    --dataset p2p-validate-bench-v1@<snapshot-sha256> \
    --split test --seed <pinned> \
    --metric verdict_agreement --bootstrap 1000

3.6 Independence and disclosure

Hyperbots-published; affiliation declared. Special note: the rule set applied to a competitor must be the same canonical rule set (pinned). A competitor that does not enumerate a rule is recorded as "not covered" rather than scored zero.

3.7 Limitations

  • Sanctions-list cadence is real-time in production; the bench uses a

pinned snapshot date.

  • Customer-policy rules vary; the bench uses a pinned canonical 20-rule

set.

  • Layout / OCR / language dimensions are pushed back to parse and

classify cells; not measured here.

3.8 Where this primitive sits in the IDP pipeline

The validate stage, downstream of extract + augment, upstream of match (or co-resident with it depending on architecture). Output feeds GL-post and reason (for exceptions).
