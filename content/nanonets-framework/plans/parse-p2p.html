Parse (P2P) — deep-study plan

1. Executive summary (TEMPLATE — numbers placeholder)

How well does an IDP system convert raw P2P document bytes — purchase orders, goods-receipt notes, supplier invoices, bills of lading, packing slips, credit notes, debit notes, freight invoices — into the canonical parse schema (header summary + line items + table cells + bounding boxes)? This study isolates the parse primitive (NOT downstream extract / match / validate) and measures it against Hyperbots IDP and [PLACEHOLDER — at least one competitor; candidates: Azure DI prebuilt-invoice, AWS Textract AnalyzeExpense, Google Document AI Invoice Parser, Nanonets invoice OCR, Rossum invoice extractor] under the di-study's 14-dimension framework, with all results currently [PENDING-MEASUREMENT].

What this study answers: per-field P/R/F1 + per-table cell P/R + TEDS, across the eight P2P doc families, stratified by complexity tier, currency, language, layout, scan condition, page count and file format.

3. Methodology

3.1 Data-status representation

Eight-row legend (Correct / Incorrect / Missed / Spurious / Structure / Assessment in progress / Vendor-reported / Not yet measured) per framework §0.4.

3.2 Sample design

  • Design parameter: n = [PLACEHOLDER] per dimension.
  • Stratified across the 55+ strata of the doc × failure-mode matrix in §5;

per-dimension target allocated across strata that exercise that dimension.

  • Macro-averaged across strata; rare-but-important strata (e.g.

hand-written GRN, Devanagari-script supplier invoice) are not outweighed by common ones (USD digital-PDF invoice).

  • Inter-annotator agreement: Cohen's κ per stratum, reported alongside

result.

3.3 Metrics

  • Field-level P/R/F1 for the parse-schema header (invoice/PO no.,

dates, party blocks, totals); macro-averaged across the field schema.

  • Line-item-level P/R/F1 for description, qty, UoM, unit price,

line total, line tax, line discount; after deterministic greedy alignment (description Jaccard + amount match) per the di-study scorer pattern.

  • Line-item count accuracy (exact match on len(line_items)).
  • Table-structure fidelity: cell-P/R against a structured GT grid +

TEDS (tree-edit-distance similarity) per the di-study §3.3.

  • Bounding-box quality: per-field IoU, reported as distribution +

median + p25/p75.

  • Reporting convention. Point estimate + 95% bootstrap CI

(≥1,000 resamples, seed pinned) + explicit n; overlapping CIs → tie-band.

3.4 Scoring rubric

  • Exact-match (casefold + whitespace-collapsed) for identifiers and

free-text fields.

  • Numeric-tolerance max(1% relative, 0.01 absolute) for monetary

fields, pinned in PINS.txt.

  • Date-tolerance: same ISO-8601 date after locale-normalisation

(e.g. 12/05/2026 → [PLACEHOLDER — pin locale rule]).

  • Wrong value = FP AND FN (silence is not a free pass).
  • Structure scored independently for tables (row segmentation, column

assignment, cell content).

3.5 Evaluation controls

  • Frozen dataset snapshot identified by sha256.
  • Rerunnable command:

`` run --study parse-p2p-v0 \ --dimension D<id> \ --system hyperbots-idp@<model-revision> \ --system <competitor>@<api-version> \ --dataset p2p-parse-bench-v1@<snapshot-sha256> \ --split test --seed <pinned> \ --metric field_f1 --bootstrap 1000 ``

  • Contamination review per document (source, license, public-availability

date, n-gram overlap) before any cell is reported.

3.6 Independence and disclosure

Hyperbots-published; affiliation declared. Protocol/metrics/scoring identical for all systems. Competitor assessed using GA offerings without deliberate misconfiguration. No measured comparative figure ships without independent methodology review.

3.7 Limitations

  • Affiliation (mitigated by §3.6 controls).
  • Managed-service drift: competitor API frozen at a stated version + date.
  • GRN ground-truth quality is workflow-dependent (hand-marked deltas);

expect wider per-field CIs on GRN strata.

  • BoL multi-leg shipments may have ambiguous line-item boundaries — flagged

in §5 as a known-fuzzy stratum, not hidden.

3.8 Where this primitive sits in the IDP pipeline

This study scopes specifically to the parse stage of parse → split → classify → extract → augment → validate → match → GL → reason. Upstream stages (none) and downstream stages (split, classify, extract, …) are not measured here.
