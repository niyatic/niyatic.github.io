Match (O2C) — deep-study plan

1. Executive summary (TEMPLATE — numbers placeholder)

The signature O2C operation: cash application = tie a remittance advice + check (or ACH/wire) to one or more open invoices, with correct apportionment of applied_amount, recognition of deductions (short-pay reasons), over-payments, on-account posting, and multi-currency conversion. Distinct from P2P 3-way match because the match is payment ↔ many invoices (one-to-many), not a triple. Numbers [PENDING-MEASUREMENT].

2. Scope & coverage

  • Workload. Remittance advice + check ↔ open-invoice subset; with

short-pays, deductions, over-payments, on-account, cross-currency remittances, lockbox + ACH + wire channels.

  • Platforms. Hyperbots match / reconcile + ≥1 competitor.
  • Coverage dims. 8 of 14.
  • Test surface.
  • n = [PLACEHOLDER] payments per dim.
  • Stratification: payment channel (lockbox / ACH / wire / credit-card),

invoice-count band (1 / 2-5 / 6-20 / 20+), short-pay deduction-code band, cross-FX, over-payment, on-account residual.

3. Methodology

3.1 Data-status representation

Eight-row legend. Per-application-line scoring: each (payment ↔ invoice, applied_amount, deduction_code) triple is a scoring unit.

3.2 Sample design

n = [PLACEHOLDER] payments per dim. Two annotators per payment; adjudication on disagreements. κ reported.

3.3 Metrics

  • Per-application-line P/R/F1.
  • End-to-end cash-application rate = fraction of payments whose

full application (all open-invoice-lines + residuals) is Correct.

  • Dollar coverage = sum(payment_amount of fully-applied payments)

÷ sum(all payment_amount).

  • Deduction-code agreement = where the system posts a deduction,

does it cite the same canonical code as the human?

  • Residual-routing correctness = if there is a residual (on-account

/ over-pay / short-pay with no code), does the system route it to the right GL bucket?

  • False-block rate.
  • 95% bootstrap CI + n.

3.4 Scoring rubric

  • Applied-amount tolerance: max([PLACEHOLDER]% rel, [PLACEHOLDER] abs)

per pinned policy.

  • FX: pinned snapshot date for the FX table.
  • Deduction-code canonical set (10 codes) pinned in PINS.txt.
  • A correct payment-to-invoice match with wrong applied_amount is

Incorrect at the application-line level.

3.6 Independence and disclosure

Hyperbots-published; affiliation declared; identical pinned policy across systems.

3.7 Limitations

  • Open-invoice subset ground-truth is customer-specific; partner-bench

tier required for high-fidelity strata.

  • Deduction-code taxonomy is customer-specific in real deployments;

the 10-code canonical set is a pinned approximation.

3.8 Where this primitive sits in the IDP pipeline

match stage, downstream of parse / classify / extract / augment / validate. Output feeds GL-post and reason (for exception routing).
