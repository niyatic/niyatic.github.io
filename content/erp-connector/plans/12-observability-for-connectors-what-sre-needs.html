Observability for connectors: what an SRE actually needs to see — plan

1. Headline + hook

Hook: "You don't get paged when an ERP integration posts the wrong thing. You get paged 11 days later when the closing fails. SRE for connectors isn't 'add Prometheus metrics' — it's 'the run object is the metric, and the dead-letter is the alert.' Here is what the connector framework emits, and what an on-call engineer actually needs from each field."

This is the depth anchor #2 — high evidence floor like blog #5. The observability surface is on disk; the post unpacks it for an SRE audience.

2. Audience + intent

  • Primary: SRE / platform engineers; secondary: finance-platform leads.
  • Reader takeaway: *"the six sinks + the lineage stamp + the exit-code

gate is the observability contract; nothing extra to wire."*

  • CTA: subscribe to the depth series.

3. The two-layer pitch (REQUIRED)

3.1 Speed

  • No extra observability wiring; the framework emits the surface.
  • [PLACEHOLDER, ~0 hours of agent observability work] vs [PLACEHOLDER,

~hours-to-days of bespoke metric / logging wiring per connector].

3.2 Depth — the observability surface

  • Six typed sinks. posted, dead_letter, unresolved,

skipped_dup, failed, audit. Each is a counter the SRE reads directly; no extra metric library.

  • The count invariant. count(sources) == sum(counts(sinks)) or

non-zero exit. The simplest possible page-worthy signal: did the process exit non-zero?

  • RunResult is the structured event. A connector run produces one

RunResult; the SRE ingests it as JSON.

  • Lineage stamp on every record. Per v0.4 README: lineage carries

target, schema_version, sign, grain, fx_policy, date_calendar. The forensic trail is already there.

  • dead_letter is the alert, with the raw response attached. Not a

generic error message; the actual response from the target. SRE triages without re-running.

  • No HITL. Low-confidence records dead-letter; an SRE never owns a

human-review queue.

  • Secrets boundary: nothing logged. secrets.py enforces no token

in logs, no token in payload, no token committed.

What the post does NOT claim

  • Built-in metric exporters (Prometheus, OTel). Not in v0.
  • Distributed tracing. Not in v0.
  • Built-in dashboards. Not in v0.
  • A scheduler / orchestrator that surfaces these signals. Per

docs/LIMITS.md, the framework is an exit-code-gated unit of work; an external scheduler integrates the signals.
