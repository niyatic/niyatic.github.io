Dynamics 365 + GP + BC: three ERPs, one agent surface — plan

1. Headline + hook

Hook: "A mid-market Microsoft shop usually has more than one Dynamics-shaped ERP in the building — D365 Business Central in one subsidiary, Dynamics GP at a legacy plant, occasionally D365 Finance & Operations in the new acquisition. Three back-ends, one agent surface."

Honest framing: as of today the connector framework ships a compiled D365 Business Central descriptor. The post pitches the polymorphic descriptor pattern — adding GP / F&O is descriptor work, not SDK work — not the claim that all three are already implemented.

2. Audience + intent

  • Primary: IT directors in mid-market Microsoft shops; secondary: M&A

integration leads.

  • Reader takeaway: *"one connector contract spans three back-ends; the

agent picks the right descriptor; you don't run three integration projects."*

  • CTA: walk-through of the polymorphic-descriptor pattern.

3. The two-layer pitch (REQUIRED)

3.1 Speed

  • For D365 BC (already on disk): agent picks target="d365_bc",

v2.0 API path, Azure AD client-credentials flow via TokenStore, BusinessKey(["companyId", "vendorId", "invoiceNumber"]). [PLACEHOLDER-time, ~hours] from descriptor pick to first mocked end-to-end.

  • For GP / F&O: descriptor authoring (~1 day each per descriptor; not

hours, name it honestly) plus agent config (~hours).

3.2 Depth — the polymorphic descriptor pattern

  • Three back-ends, three descriptors, one normalizer family (the

purchaseInvoices / apInvoice shape).

  • Master-key resolution differs (BC's vendorId GUID vs GP's VENDORID

string vs F&O's dataAreaId+VendAccount); the resolver lambda absorbs the difference.

  • Idempotency lined up to each back-end's own uniqueness: BC's

invoiceNumber, GP's VCHRNMBR, F&O's Voucher+AccountingDate.

  • Secrets boundary common (Azure AD), with per-tenant realmId /

companyId salting the byte-hash.
