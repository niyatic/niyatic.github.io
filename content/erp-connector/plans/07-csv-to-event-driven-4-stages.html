From CSV to event-driven: an evolving integration in 4 stages — plan

1. Headline + hook

Hook: "Real integrations evolve. You start by dropping CSVs in a folder because that's what the ERP team will accept this quarter. Six months later you're posting to an API. A year in, you want batches with back-pressure. Eventually, you want webhooks and exactly-once replay. With the connector framework, each stage is hours of agent work — and the artifact at each stage is byte-identical when re-run."

The post is the journey narrative — time-compressed; it threads the two-layer pitch through a multi-stage adoption arc.

Honest framing: framework v0 ships the FileSink (stage 1) and ApiSink (stage 2). Stage 3 (batched + back-pressure) is partially covered by the ApiSink's "batch cap, Retry-After honored, jittered backoff" surface; stage 4 (event-driven webhooks with replay) is roadmap, recorded in plan/PLAN-next-steps.md. The post must mark stage 4 as roadmap.

3. The two-layer pitch (REQUIRED)

3.1 Speed — stage by stage

  • Stage 1: CSV emit (live today). Agent picks FileSink, drops to

the ERP team's intake folder. [PLACEHOLDER, ~1-2 hours.]

  • Stage 2: API post (live today). Agent swaps FileSink for

ApiSink; same descriptor, same normalizer, same idempotency key. Re-run is a no-op. [PLACEHOLDER, ~2-4 hours.]

  • Stage 3: Batched + back-pressure (live today). ApiSink's batch cap

+ Retry-After + bounded retry → dead-letter. [PLACEHOLDER, ~hours.] Mostly config, not code.

  • Stage 4: Event-driven webhooks with replay (ROADMAP). Plan

surfaces it; framework lands the webhook-source pattern with the same SourceBytesKey contract so a re-delivered webhook is a skipped_dup. [PLACEHOLDER pending framework v1.]

3.2 Depth

  • One descriptor, four sinks. The descriptor changes minimally

across stages — schema_version, master_key resolution, sign / FX / grain stay the same. The sink (and the source, at stage 4) is what changes.

  • Same idempotency key across stages. Byte-hash default; the same

CSV bytes re-ingested at stage 2 dedups against the stage-1 record.

  • Same conformance kit. Replay-safety, count invariant, lineage —

every stage passes the same suite.

  • Same exit-code gate. Stage 4 webhooks honor the same gate.
