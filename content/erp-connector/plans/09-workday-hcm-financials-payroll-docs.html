Workday HCM + Financials: connecting payroll docs without losing context — plan

1. Headline + hook

Hook: "Payroll documents don't live in one Workday module — they span HCM (the employee, the cost center, the manager) and Financials (the GL mapping, the period, the company). A bespoke integration usually picks one side and writes glue for the other. The connector framework's descriptor pattern carries the cross-module context as policy, not glue."

Honest framing: as of today the connector framework does not ship a Workday descriptor. This is a framework-extension post — same caveat as blog #4 (Oracle). The two-layer pitch works because the pattern is what sells; the speed claim is deferred until the descriptor lands.

3. The two-layer pitch (REQUIRED)

3.1 Speed (claim deferred)

Until a Workday descriptor exists, no wall-clock number is named. The post can describe the shape of the agent's work — pick the HCM lookup, pick the Financials descriptor, wire the cross-module resolver — without a speed headline.

3.2 Depth

  • Cross-module master key. master_key_resolver returns a tuple:

(employee_id_HCM, cost_center_HCM, gl_account_Financials). One lookup call, one policy.

  • Sign / FX / grain per module. Payroll postings to Financials have

a fixed sign convention per company; descriptor encodes it.

  • **Idempotency keyed to the event (pay-run + employee + period),

not the file.** BusinessKey(["pay_run_id", "employee_id", "period"]) lines up with Workday Financials' own uniqueness.

  • No HITL on low-confidence employee match. Low-confidence resolver

output dead-letters with the raw lookup response.
