Stop assuming a document is signed: which state is it, in one call

Contract and AP workflows stall when no one knows whether a document is executed. An unsigned master agreement is not enforceable; wet ink and a cryptographic certificate carry different assurance; a stamp may not bind. hyperapi-classify-signature-presence turns that triage into one function call that returns a typed state with a confidence score on every option.

The hard part isn't OCR

It's that documents rhyme. The library surfaces the service's per-class confidence verbatim and runs a deterministic channel_consistent() check -- every class belongs to exactly one execution mode (none / manual / electronic / cryptographic), so a cross-mode flip is caught even when the score looked clean. Classes that share one execution mode are caught by the runner-up margin gate instead.

No queue, just a flag

A borderline first pass -- low top score *or* a thin margin to the runner-up *or* a execution mode conflict -- is automatically re-classified on a higher-resolution profile; the stronger pass is kept. Documents that stay weak come back trustworthy=false for your automation to route by rule. Routing stays in your automation, end to end.

Five lines

``python from hyperapi_classify_signature_presence import classify_file r = classify_file("doc.pdf") print(r.classification.doc_class, r.trustworthy) ``

Accuracy is published at apis.hyperbots.com. The classification model is an open-source document model fine-tuned by Hyperbots (Hyperbots IP); this repo is the typed client around it.
