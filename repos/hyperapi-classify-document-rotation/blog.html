The cheapest OCR accuracy win: fix the rotation first

Every finance-document pipeline has a silent accuracy leak nobody profiles: scans that arrive sideways or upside-down. A page rotated 90° doesn't OCR badly — it OCRs *wrong*, and the error propagates through extraction, classification, and matching as if the data were real.

One pre-flight question

hyperapi-classify-document-rotation answers exactly one question, per page: which way is up? It returns the detected rotation (0 / 90 / 180 / 270) and the exact clockwise angle to apply to correct it, as typed JSON with a confidence score on every page. Feed the corrections to your PDF library, then run the rest of your pipeline on upright pages.

No queue, just a flag

A weak first pass is automatically re-classified on a higher-resolution profile; the stronger pass is kept. Scans that stay weak come back trustworthy=false with the weak pages named, for your automation to route by rule. Routing stays in your automation, end to end.

Two lines

``python from hyperapi_classify_document_rotation import classify_file r = classify_file("scan.pdf") print(r.to_dict()["corrections"], r.trustworthy) ``

Accuracy is published at apis.hyperbots.com. The orientation model is an open-source document model fine-tuned by Hyperbots (Hyperbots IP); this repo is the typed client around it.
