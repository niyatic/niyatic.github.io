2-week release plan: v0.4 + first paid pilot

Target: ship v0.4 on Sat 2026-06-07. Real-customer benchmark on at least one ledger. Public-readable repo by Sun 2026-06-08.

Week 1 · Mon May 25 — Sat May 31

Engineering track

  • Mon: Run /qa-full on v0.3.0. Fix anything BLOCK before any other engineering work.
  • Mon-Tue: Run pharma persona test (write a synthesized $1B specialty-pharma close pack; same 3 tasks). Document gaps.
  • Tue-Wed: Run /qa-vertical on the 8 sub-industries that have never been tested individually (banking-commercial, energy-utilities-regulated, healthcare-rcm, insurance-pc, insurance-health, logistics-3pl, mortgage-secondary, distribution-wholesale).
  • Wed: Rebase against NanoClaw trunk. /qa-adversarial on the 3 v0.3.0 companion files.
  • Thu-Fri: Author 3 v0.4 methodology items (the ones most likely to come up in pilot calls: HGB→IFRS, hedge accounting, gift-card breakage). Same review chain.
  • Sat: Tag v0.3.1 (intermediate, internal) once /qa-full passes clean.

CFO / domain track

  • Mon: Pick 3 design partners from existing Hyperbots customers (1 mfr, 1 retail, 1 pharma or healthcare-RCM). Send signed-NDA + anonymized-close-pack request.
  • Tue-Wed: Receive anonymized close packs (trial balance, plant/store P&L, lease portfolio for retailer, batch records for pharma).
  • Wed: Map each partner's actual close workflow against the 16-skill catalogue. List skill gaps per partner.
  • Thu: Write a 1-page test protocol per partner — what we will run, what we will measure, what we expect to get wrong.
  • Fri: Internal dry-run of the v0.3.0 stack on one partner's data. Eng + finance domain reviewer together. Decide if pilot is ready.
  • Sat: Sign-off go/no-go on Monday pilot starts.

Week 2 · Mon Jun 2 — Sat Jun 7

Engineering track

  • Mon-Wed: Run the pipeline against each of the 3 partner ledgers. Capture every output. Stream stage-level metrics into a per-partner report.
  • Wed-Thu: Fix the issues that come up in real-data running. These will be different from synthetic-data issues. Schema edge cases, weird vendor names, multi-period documents.
  • Thu: Build the public README + quickstart. git clone → claude code . → /qa-vertical fashion-retail in <5 min.
  • Fri: Final /qa-full. Tag v0.4.0. Flip GitHub + HF repos public (if approved).
  • Sat: Push v0.4.0 tag. Write release notes. Update changelog.

CFO / domain track

  • Mon-Wed: Walk through each partner's actual output with their CFO. Two-hour session per partner. Capture verbatim "this is wrong because…" feedback.
  • Wed-Thu: Categorize feedback into (a) eng bugs, (b) skill content gaps, (c) genuinely-novel domain edge cases. Route each to the right track.
  • Thu-Fri: Write 3 customer case studies (anonymized): the close pack, the variance commentary, the covenant check. These become the v0.4.0 launch material.
  • Fri: Sales 1-pager. Two CFO testimonial quotes if any partner will go on the record.
  • Sat: Final review with each partner. Confirm anonymization, confirm permission to publish numbers.

What gets released on Sat Jun 7

  • Public GitHub + HF Hub at v0.4.0 — 16 base skills + 3 v0.3.0 companions + 3 v0.4 methodology items + 8 newly-tested sub-industries.
  • Three customer case studies (anonymized) — close pack, variance commentary, covenant check.
  • One sales 1-pager.
  • Public quickstart (clone → load skills → run /qa-vertical in <5 min).
This is a forcing function. If we cannot get 3 design partners by Wed of week 1, we slip to a 3-week plan and ship without real-customer benchmark. That is a separate decision.

Risks the plan does not cover

  • Partner data hygiene. Real ledgers contain PII / vendor names / customer names. Anonymization is the partner's responsibility per the NDA. If they send raw data we have a problem.
  • Methodology disagreements. Two CFOs may have legitimately different opinions on the same GAAP question (e.g., how to split markdown reserve between in-period and catch-up). The plan assumes we document both, not arbitrate.
  • NanoClaw trunk could move significantly in the 2-week window and force a re-rebase mid-cycle. Probability low; impact 1-2 days.
