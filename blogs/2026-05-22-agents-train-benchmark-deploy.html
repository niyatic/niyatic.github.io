Our Agents train, benchmark and deploy all of that while we sleep, under $20

We build document-AI for finance. This post is about the thing *behind* the

product: the agent system we run overnight to do machine-learning research on

itself — survey the literature, analyze data, stand up an environment, run a

training or evaluation pass, benchmark it, write up what happened, and propose

the next experiment. The honest version of that story is more interesting than

the headline, so let's start with the honest version.

The honest hook

The most useful result our overnight agent runs produced this month was a

*negative* one. We pointed our reproducibility harness at 23 benchmark cells we

had previously measured in-house, fully expecting it to re-derive them. It

re-derived 0 of 23 — not because the numbers were wrong, but because the

offline harness structurally cannot reach the hosted models and the in-house

corpus those cells depend on. The agents recorded that, named the three

structural causes per cell, retracted nothing, and fabricated nothing.

That is the bar this post is written to. An agent system that "trains and

benchmarks while we sleep" is only worth describing if it also tells you, in

the morning, exactly what it could *not* establish. So: here is the

architecture, here is what one run actually outputs, and here is — precisely —

where it stands today.

What the system is

The reference architecture is an Autonomous ML Research Engineer Agent — a

single Claude-powered Master Agent acting as a Run Orchestrator, delegating to a

roster of specialized subagents over a shared tooling layer, iterating around a

research-lifecycle loop. Concretely, in our DocumentAI Benchmark Factory, that

Master Agent coordinates a 9-agent roster:

  • frontier-lab-cto — sets the scientific bar and holds STOP authority on

any unsound result.

  • benchmark-product-manager — owns the leaderboard as a product: task

taxonomy, the model-submission contract, refresh cadence.

  • dataset-curator — the dataset registry: provenance, license, frozen splits,

sha256 manifests.

  • model-wrangler — model adapters and pinned revisions, with per-model

cost/throughput fields.

  • eval-harness-engineer — the pinned, deterministic, fair eval harness.
  • contamination-auditor — train/test leakage and held-out integrity.
  • stats-methodologist — canonical metrics, confidence intervals, tie-bands.
  • qa-reproducibility — independently re-runs sampled cells; **blocks on

mismatch**.

  • leaderboard-frontend-eng — regenerates the static public leaderboard.

The roster maps cleanly onto the reference architecture's subagents. The

Survey/Literature, Data Analysis, Setup/Environment, Training, Evaluation/

Benchmarking, Discussion/Error-Analysis and Reporting/Documentation roles are

the *functions*; the 9 named agents are how we've instantiated them for a

finance-document-AI benchmark. The Master Agent's job — Planner, Task

Decomposer, Context Builder, Subagent Router, Decision/Approval Logic, Run

Memory Manager — is to turn one objective ("ship a defensible leaderboard cell")

into a decomposed plan, route each step to the right subagent, and carry the

run's memory forward.

The architecture, section by section

The Run Orchestrator (Master Agent)

The orchestrator never produces a number itself. It plans, decomposes, builds

context, routes, and decides. Its Decision/Approval Logic is deliberately

conservative: irreproducible entries are rejected, named competitors are scoped

to the governed benchmark artifact only, and anything that would exceed an

approved compute budget triggers the CTO agent's STOP. Its **Run Memory

Manager** is what makes an overnight run auditable in the morning — every

decision and every blocker is written down, not just the wins.

The specialized subagents

Each subagent owns one slice end-to-end:

  • Survey / Literature — grounds a task in prior work and existing task

catalogs before any code is written.

  • Data Analysis — inspects and registers datasets: provenance, license,

split, content sha256. Splits are frozen at registration so the eval set

cannot drift.

  • Setup / Environment — pins the world: Python version, every dependency at

an exact version, the run seed. Our seed harness is **stdlib-only and

offline** specifically so it reproduces bit-for-bit on a stranger's machine.

  • Training / Evaluation — runs an adapter (a pure

predict(prompt, decoding) -> text function) against a task×dataset and

emits a metric, a 95% CI, an n, and the literal rerun command.

  • Discussion / Error Analysis — turns outliers into audits, not silent

reruns-until-good.

## Sources and ## Open assumptions.

Per-run Training Session States

Every run snapshots its own state: the environment snapshot and dependency

pins, the data version (sha256-frozen), the config/hyperparameters and pinned

decoding, the logs, the metrics, and the artifacts/checkpoints. This is what

lets a later agent — or a later human — re-open a run and see exactly the world

it executed in. A leaderboard cell that cannot point back to its session state

does not ship.

The Tooling / Execution layer

The subagents act through a shared layer: Git for versioning, a dataset/feature

registry, Python/CLI for execution, compute for training and inference, an

experiment tracker, an artifact/model store, web/paper search for the survey

step, and an issue tracker for blockers. In our seed configuration this layer is

deliberately minimal — offline, stdlib-only — and it scales up by *adding

registry entries and adapters*, not by rewriting the harness.

The end-to-end loop

A run is one turn of an iterative lifecycle:

**Objective → Survey → Data Analysis → Setup → Training/Evaluation →

Discussion → Updated Plan → (next Objective)**

The orchestrator sets an objective, the Survey subagent grounds it, Data

Analysis registers and freezes the data, Setup pins the environment, the

Training/Evaluation subagent produces measured results, Discussion analyzes

what happened and what failed, and the loop closes with an updated plan that

becomes the next objective. Crucially, the loop can close on a *negative*

finding — "this could not be established, here is why" — and that is treated as

a completed turn, not a failure.

What a run actually outputs

Concretely, here is what one overnight run of the Benchmark Factory produced —

all of it verifiable in the repository:

  • An implementation plan + experiment reports. The leaderboard spec, the

scale-to-100 plan, and a per-workstream audit log.

  • Benchmarks. A SEED leaderboard of 4 tasks (line-item extraction,

payment-terms extraction, totals validation, document-type classification)

× 1 dataset × 3 models = 12 real measured cells. Each cell is offline,

stdlib-only, deterministic, pinned to seed 20260516 and a recorded git

commit, and carries its own literal rerun command.

  • Trained models / checkpoints. Here we are precise: the *seed* runs

evaluate models, they do not train them. The fine-tuned models in our

benchmark — open-source backbones fine-tuned by Hyperbots (Hyperbots IP),

via LoRA — were trained in a separate, in-house workstream, and those numbers

are *attested, not factory-reproduced* (more on this below).

  • Eval dashboards. A single static HTML leaderboard, no build step, no

network, with per-cell drill-down to the exact rerun command, the 95% CI, the

n, and the contamination record.

  • Insights / next actions. Including the negative finding that opened this

post — recorded per-cell, with the three structural causes named.

And a reproducibility gate sits across all of it: harness/repro_check.py

re-runs at least one sampled cell per release and **STOPs the run on any

mismatch**. A contamination audit (contamination/audit.py) attaches a record

per (model, dataset) before any cell is allowed to ship.

About "$20" and "while we sleep"

The title makes two claims, and we want to be exact about both.

"While we sleep" is real in the sense that matters: the agent roster runs

the full survey → data → setup → evaluate → discuss → re-plan loop unattended,

overnight, and the orchestrator's Run Memory Manager makes the result auditable

the next morning. That is the operating model.

The cost figure needs care. The repository does not substantiate a

single "$20 per overnight run" total, so we will not assert one. What it *does*

substantiate is a real, infrastructure-derived inference cost: on an A100

40 GB at roughly $2/hour, the fine-tuned SAVIOR-Train inference path measured

$0.000186 per attempted call, $0.000191 per successful call, and

≈ $0.19 per 1,000 successful calls. Separately, the SEED leaderboard runs

are offline and stdlib-only — they require near-zero compute and no API

spend at all, because there is no network egress and no GPU in that path.

So the honest framing is: the *seed evaluation* loop is effectively free to

run, and the *measured inference* cost of the fine-tuned model is fractions of

a cent per document. A consolidated, end-to-end "cost per overnight run" number

that rolls in fine-tuning compute, hosted-API inference, and orchestration is

not yet computed in the repository, and we are not going to invent one.

Where this stands — honestly

Three things are true at once, and a credible post states all three.

1. The seed leaderboard is real and reproducible. 12 measured cells,

offline, stdlib-only, deterministic, pinned, with a STOP-on-mismatch repro

gate and a contamination audit. A stranger can re-run it.

2. **The headline benchmark numbers are *attested*, not factory-reproduced.**

The 23-row DocumentAI parse-intent table — and the broader SAVIOR-Bench

results — were run in-house and user-attested (2026-05-18). When our

overnight harness tried to independently re-derive those 23 cells, it

re-derived 0 of 23: the offline seed harness has no adapters for the

hosted vendor models, no loader for the in-house corpus, and the attested

source carries only aggregate scores (no per-document vector, so no honest

bootstrap CI). We documented that per cell, preserved every attested value

unchanged, retracted nothing, and added no fabricated confidence interval.

This is not an overclaim we are hiding — it is the lead finding.

3. Fine-tuning bought a narrow edge, disclosed as such. Where we compare a

Hyperbots-IP fine-tuned model against zero-shot systems, those rows are

not apples-to-apples — the fine-tuned rows are LoRA fine-tunes on

in-domain financial documents; the baselines are zero-shot in native

settings — and we label that inline, never as a footnote. Benchmark figures

here are model OCR/parse performance, not a product claim; any statement

about the Hyperbots product's accuracy is referenced only "per

apis.hyperbots.com".

The system genuinely does run the research loop overnight, cheaply, and writes

down what it found. The asset is not a hype number — it is that, the next

morning, you can trust the report, including the parts that say "we could not

establish this yet."
