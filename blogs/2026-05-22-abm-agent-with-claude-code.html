We Built Our Entire ABM Engine in Claude Code — Here's What That Actually Looks Like

A few years ago, standing up an account-based marketing motion meant a contractor,

a stack of point tools, and six weeks before the first email went out.

We did it differently. Our entire ABM engine — account selection, research,

committee mapping, per-account decks, outreach sequences, a deploy pipeline — was

built inside Claude Code as a team of 14 agents. No ABM SaaS subscription. No

agency. The whole thing lives in one git repository, and every artifact it

produces is reviewed and merged by a human before anything reaches a prospect.

I want to walk through it honestly, because the interesting part is not "AI did

our marketing." It didn't. The interesting part is what it actually looks like

when you treat a go-to-market function as software — version-controlled,

test-gated, and built by agents with explicit, narrow jobs.

One thing up front, because it matters: **we have not sent a single message

yet.** Every deck and every sequence in this system is a draft sitting behind a

human-approval gate. So this is a build story, not a results story. I'm not going

to quote you open rates we haven't measured.

What we built

Fourteen agents, each with one job, coordinating through git branches and pull

requests instead of a shared inbox.

Five of them are the core line: abm-sales defines the ideal customer profile,

scores and tiers accounts, and maps buying committees. abm-marketing turns

that into per-account messaging and channel plans. abm-dev-manager tracks

status and enforces the workflow. abm-qa reviews *every* pull request against

a brand-guardrail checklist. abm-manager-approver gives the final GO / NO-GO.

Five more handle execution depth: an account-executive agent that writes the

decision-grade deck narrative, a solution-architect that fills in the

technical pitch, a gtm-lead that engineers the outreach sequence, a

content-marketing agent that certifies copy against a conversion bar, and a

VP agent that issues an APPROVED-or-REWORK verdict on each deck.

The last four are specialists: a deploy-engineer for the static site and

Slack/Teams notifications, a clay-engineer for enrichment and CRM handoff, an

events-discovery agent, and a b2c-virality agent for community-led growth.

They run as a sequential pipeline — sales → marketing → account-executive →

solution-architect → gtm-lead → content-marketing → VP → QA → human merge. The

baton between stages is a pull request on a feature branch. Nothing is "done"

until QA passes it and a human merges it.

How it was built

I opened Claude Code and described the system at a high level: a small ABM team,

each role its own agent, work moving through pull requests, a human gate at the

end.

What came back was not just code. It was a decomposition. Claude Code proposed

the role split, suggested the branch-and-PR workflow as the coordination

mechanism, and — the part that changed how I thought about it — proposed a

*separate, independent QA agent* whose only job is to try to fail every pull

request. I had described a sales-and-marketing team. It told me I also needed an

adversary in the loop. It was right.

The build was phased, and the git history shows it cleanly across 194 commits,

all from this month. First the agent-and-workflow scaffold. Then the deploy

infrastructure — a static-site build, basic-auth, and a Slack/Teams notifier.

Then the long stretch: account enrichment at scale, batch after batch. Then the

extended agent team. Then dashboards and an events charter.

I wrote no application logic for the marketing itself. The longest thing I typed

was the description of our ideal customer and the guardrails the copy must never

cross.

The inputs

The system starts from a tightly-defined ICP: four document classes Hyperbots

turns into structured data — finance documents, bank documents, logistics and

shipping documents, and custom types — and a set of buyer personas across

fintech, banking, insurance, logistics, and healthcare.

From that, abm-sales built the target list. Every candidate is scored on

three axes — Fit weighted ×3, Intent ×2, Deal size ×1, each on a 1–5 scale, for a

maximum of 30. Tier-1 is a score of 24 or above, Tier-2 is 16–23.

The curated primary list is 50 accounts: 18 Tier-1, 32 Tier-2. An expansion

pass took it to 288 more, for 338 verified prospects in total — 70-plus of

them India-headquartered. Alongside the buyer list sit 10 partnership targets

and 12 ERP integration targets.

There is one rule on the target list I want to call out, because it's a guardrail,

not a preference. We call it Goal G0: **zero invoice-processing-automation

competitors anywhere on the list.** When the expansion pass surfaced five

competitor-adjacent companies, they didn't get quietly dropped — they were moved

into a labelled cohort with a mandatory build-versus-buy gate before anyone is

allowed to touch them. The system doesn't hide its edge cases. It flags them.

How the data is enriched

Enrichment is where most "AI GTM" tools quietly fabricate. Ours is built so it

can't.

For the top accounts, the system produces committee maps — 12 Tier-1 accounts

mapped to economic buyer, technical champion, and likely blockers, all by *role*,

never by named individual. The clay-engineer agent runs a Clay-style

enrichment layer: firmographic and contact-by-role waterfalls, signal triggers to

watch (funding, hiring, technology installs, leadership news), and a CRM-handoff

field mapping.

The hard rule on that agent: no fabricated contacts, emails, titles, headcounts,

funding figures, or metrics. Anything not verifiable against a public source

becomes an explicit waterfall placeholder instead of an invented value. Every

account claim in a deck cites a working public URL. If a fact can't be sourced,

it doesn't ship.

What the system produces

The output is concrete and countable.

72 account decks, each generated in three formats — markdown, PDF, and

PowerPoint — for 216 deck files in total. Every deck follows the same spine:

the account's world, the cost of the status quo, what changes, the proof, the

ask, a solution architecture, an outreach sequence, a content certification, and

a VP review — closing, like every artifact in this repo, with a ## Sources and

## Open assumptions section.

For LinkedIn, the system drafted 8 buyer-persona DM templates — CFO, VP

Finance, Controller, Procurement, IT/CIO, ERP Lead, AI/Data Strategy, and

Partnership/BD — plus 3 follow-up sequences timed to day +3, +7, and +14.

And it produces the operational layer: a static GTM site with role-specific

dashboards, a shareable bundle with the full account CSV, and Slack/Teams

digests — an account digest, a warm-path digest, and an internal-only signal

digest that is explicitly flagged "do not share with customers."

The real product is the loop

The agents are useful on their own. But the actual product is the review loop.

Here is a real example from this month. The QA agent looked at the execution-v1

delivery and returned a verdict: CHANGES REQUESTED, two defects. One of them

was those five competitor-adjacent accounts sitting in the main target list — a

Goal G0 violation.

In a normal workflow, that's a meeting and a week. Here, the verdict *is* the

instruction. The sales agent picked up the QA finding, moved the five accounts

into a labelled cohort with a pre-touch build-versus-buy gate, and put the work

back up for review. The approval state is, honestly and visibly, "GO WITH FIXES"

— blocked until QA passes it again.

That's the loop. A guardrail is written once. The QA agent enforces it on every

pull request, and a single violation is CHANGES REQUESTED — never waived. The

guardrails themselves are not soft suggestions: no competitor is ever named (only

category labels like "legacy IDP"), accuracy is only ever stated as "per

apis.hyperbots.com" and never invented, the models are described as open-source

fine-tuned by Hyperbots — Hyperbots IP — and pricing is only ever the published

$0.05-per-page model. No invented quotes. No manual-review framing.

The system is fast at the *cycle*, not just the task. That's the difference that

matters.

The pattern adapts

We built this for ABM. But the architecture — narrow agents, a PR pipeline, an

independent QA gate, a human merge — is not specific to marketing.

The same shape already spun off an events-discovery agent that builds a

conference charter across three regions, a community-growth agent, and — in a

sibling repository — a Reddit social-listening agent for our finance-and-

accounting solution areas. Same skeleton. Different job. The agents are defined

in plain English, so the cost of a new function is a description and a review

gate, not a new tool contract.

Where this actually stands

I'll end where I started, honestly. As of today the plan is approved *as a

method*. The decks are drafted and not sent. The LinkedIn sequences are written

and not posted. The notifiers are built and not yet wired to live execution.

Every external action still waits on an explicit human authorization, and the

current QA verdict is CHANGES REQUESTED.

That is not the system being incomplete. That is the system working as designed.

The win here isn't "we automated outbound." It's that a go-to-market function now

behaves like software: every claim is sourced, every guardrail is enforced by an

agent whose only job is to enforce it, and a human signs off before anything

ships. Built by 14 agents, in one repository, this month.

The expensive part of GTM used to be the building. We moved the expensive part to

where it belongs — the judgment about what's good enough to send.
