About this report

This report establishes a structured, vendor-neutral framework for comparing intelligent document-processing platforms on the task of invoice extraction, and applies that framework to two named offerings: Hyperbots Intelligent Document Processing (IDP) and Microsoft Azure Document Intelligence (Azure DI). It defines the evaluation surface — fourteen measurement dimensions, a fifty-five cell invoice / line-item combination matrix, and a crosswalk of two external metric taxonomies totalling 404 items — and sets out the measurement methodology by which comparative results will be produced.

The present edition is a framework and design release. It reports the evaluation structure and the coverage landscape; it does not report measured head-to-head accuracy. Where a comparative result has not yet been measured, the report says so explicitly through the data-status legend defined in Section 3 and applied throughout.

1. Executive Summary

Update — preliminary Mini Pilot data is now available. Since the framework edition was first issued, a small Mini Pilot has been scored against both platforms. Its results are reported in the dedicated Mini Pilot — Preliminary Results section. They are preliminary and based on a partial sample (partial n), so they are indicative only and do not replace the full study; the head-to-head edition is still pending. Readers seeking early signal should consult that section.

Invoice extraction is among the highest-volume document-automation workloads in enterprise finance operations, and the accuracy with which a platform extracts header, party, monetary and line-item fields directly governs the share of documents that can be processed without human intervention. This report addresses a recurring question from buyers evaluating that workload: on what evidence should one intelligent document-processing platform be preferred over another?

The report's contribution is a structured comparison framework rather than a verdict. It decomposes invoice-extraction performance into fourteen measurement dimensions (Section 4), enumerates a fifty-five cell matrix of invoice types and line-item characteristics that defines the test surface (Section 5), and reconciles two independent external metric taxonomies — a 200-item complexity taxonomy and a 204-row extraction inventory, 404 items in total — against those fourteen dimensions (Section 6). The framework is applied to Hyperbots IDP and Microsoft Azure Document Intelligence.

1.1 Key observations

  • The evaluation surface is broad and well-defined. Fourteen dimensions, fifty-five enumerated invoice / line-item combinations, and a 404-item taxonomy crosswalk give a comprehensive and auditable test surface for invoice extraction.
  • Five dimensions are not represented in existing industry taxonomies. Confidence calibration, aggregate throughput, visual grounding, straight-through-processing rate, and per-document latency (D9–D13) are introduced by this framework; neither external taxonomy enumerates them. This is reported as a gap in current comparison practice rather than concealed.
  • HyperAPI coverage of the invoice-complexity space is broad. HyperAPI's own capability-coverage assessment (Section 7) projects coverage of 76 of the 204 mapped complexity dimensions fully and 101 partially, with the depth frontier in script and character variation, locale-specific identifiers, and image / scan quality. This is a scope claim, not a measured result.
  • No measured comparative result is available in this edition. Head-to-head accuracy between the two platforms has not yet been measured; the report presents the framework, not the scores.

1.2 Status of this edition

This is a framework and design edition. It does not contain measured head-to-head results. Every comparative result cell carries the status “Not yet measured”. The coverage figures in Section 7 are HyperAPI's own capability-coverage assessment — a scope claim, not an independent or measured evaluation. A subsequent edition will report measured results once primary data collection is complete and the framework has been executed against both platforms.

1.3 Intended audience

This report is written for technology-evaluation and finance-operations leaders selecting an invoice-extraction platform, and for the analysts and architects who support that selection. It is structured so that the framework (Sections 2–6) can be adopted independently of any particular vendor outcome.

Section 2

2. Scope & Coverage

2.1 Workload in scope

The report is restricted to a single document workload: invoice extraction. It covers the structured extraction of header fields (invoice number, dates, purchase-order reference), party fields (vendor and buyer identity, tax identifiers), monetary fields (subtotal, tax, total), remittance fields, and the line-item table. Adjacent workloads — purchase-order matching, receipt processing, contract analysis — are out of scope for this edition.

2.2 Platforms and capabilities assessed

Two platforms are assessed. For each, three capability tiers are considered, so that the comparison is made at equivalent capability levels:

Prebuilt invoice extraction Out-of-the-box invoice field extraction. For Azure DI this is the Prebuilt Invoice model; Hyperbots IDP is assessed on its corresponding prebuilt invoice configuration.
Layout / table extraction Text, selection-mark and table-structure extraction — the basis for layout-aware comparison. For Azure DI this is the Layout API; Hyperbots IDP is assessed on its corresponding layout/table configuration.
Custom extraction A customer-trained extraction model, assessed under a documented, equal training budget for both platforms.

Exact product SKUs, API versions and model revisions are recorded alongside every result cell at the time of measurement.

2.3 Coverage dimensions

Platform performance is decomposed into fourteen measurement dimensions (D1–D14), detailed in Section 4: document complexity, field coverage, document layout variance, currency coverage, language coverage, context coverage, OCR / scan degradation, table / line-item structure fidelity, field confidence calibration, aggregate throughput, visual grounding, straight-through processing, time to process, and file-format coverage. The dimensions are designed to be non-overlapping; in particular, aggregate throughput (D10, a fleet-level sustained-capacity measure) and time to process (D13, a per-document latency measure) are kept distinct.

2.4 Test surface

The test surface over which the dimensions are evaluated is enumerated as a fifty-five cell invoice / line-item combination matrix (Section 5). Each cell pairs an invoice type with a line-item characteristic and a layout family, and identifies the dimensions it most directly exercises.

2.5 Boundaries of this edition

  • The report covers the framework and the coverage landscape. Measured comparative accuracy is out of scope for this edition and is reserved for a subsequent measured edition.
  • Section 7 reports HyperAPI's own capability-coverage assessment across the 204-dimension invoice-complexity space; it is a scope claim, not an independent audit or a measured benchmark.
  • Out-of-distribution invoice templates not represented in the fifty-five cell matrix are out of scope for this edition of the framework.

3. Methodology

This section defines how comparative results will be produced when the framework is executed, and how the report represents results that have not yet been measured.

3.1 Data-status representation

Comparative result columns in this report are not left blank and are not filled with placeholder numbers. Each cell carries an explicit data-status label drawn from a fixed legend, so that a reader can tell at a glance whether a cell represents a measured result, a measurement in progress, a vendor-reported figure, or a measurement not yet undertaken. The legend is reproduced below and applies to every result column in Sections 4 and 7.

Data-status legend. Every cell in a comparative result column carries one of the status labels below. As of this edition, no comparative result column has been populated with a measured value; all such cells read “Not yet measured”.
Status labelMeaning
Not yet measured No measurement has been collected for this cell. No comparative result exists.
Assessment in progress Data collection for this cell is underway but not yet complete; no result is reported until it concludes and passes review.
Vendor-reported A value supplied by a vendor as a self-assessment or self-classification. It is reproduced for context only and is not an independent or measured result.
Interpretation. Throughout this edition every comparative result cell carries the status “Not yet measured”. The report therefore presents no measured comparative accuracy between the two platforms. A status label is a statement about data availability; it is never a proxy for, and must not be read as, a result.

3.2 Sample design

The measurement protocol provisions a target of 5,000 tagged invoice documents per dimension, drawn with stratification so that each sub-category — each layout family, currency, degradation level and input file format — receives a balanced share. This figure describes the planned corpus for the full measured edition; it is a design parameter and not a count of documents already evaluated.

3.3 Metrics

  • Field-level precision, recall and F1 — per-field, macro-averaged across the invoice schema. Exact-match for identifiers and dates; numeric-tolerance match for monetary fields.
  • Layout-aware extraction accuracy — correctness conditioned on layout family and on spatial reading-order reconstruction.
  • Table / line-item structure fidelity — cell-level precision and recall against a structured ground-truth grid, with a tree-edit-distance similarity for table structure.
  • Reporting convention — each measured metric will be reported as a point estimate with a 95% confidence interval and an explicit sample size n. Where two platforms' confidence intervals overlap, the result will be reported as a tie band rather than a strict ranking.

3.4 Scoring rubric

CorrectExtracted value matches ground truth under the field's match rule (exact, numeric-tolerance, or normalized).
IncorrectValue present but does not match.
MissedGround-truth field present, no value extracted (a recall loss).
SpuriousValue extracted for a field absent in ground truth (a precision loss).
StructureFor tables, row segmentation, column assignment and cell content are scored independently.

Ground truth is double-annotated, with disagreements adjudicated by a third reviewer; inter-annotator agreement is reported with the dataset.

3.5 Evaluation controls

  • Development and test splits are frozen at dataset registration; the test split is not used for prompt or model tuning.
  • Any custom-extraction model is trained only on the development split, under a documented and equal training budget for both platforms.
  • A contamination review records each document's source, license and public-availability date and runs an overlap check before a cell is reported.
  • Each measured cell is associated with the parameters that produced it, so the measurement can be reproduced. The command form is:
run --study azure-di-invoice-v0 \ --dimension D3 \ --system hyperbots-idp@<model-revision> \ --system azure-di-prebuilt-invoice@<api-version> \ --dataset invoice-bench-v1@<snapshot-sha256> \ --split test --seed 20260522 \ --metric layout_aware_f1 --bootstrap 1000

3.6 Independence and disclosure

This report is published by Hyperbots, an affiliate of the vendor of one of the two platforms assessed. This is a material affiliation and is disclosed prominently. The framework is designed to constrain the effect of that affiliation: the protocol, metrics and scoring rules are fixed and identical for both platforms; comparative results are produced only by the documented measurement procedure; and Azure DI is assessed using its generally available offerings without deliberate misconfiguration. No measured comparative figure appears in this edition, and none will be added to a future edition until the measurement has been independently reviewed against this methodology.

3.7 Limitations

  • Affiliation. The report is published by an affiliate of one platform's vendor; the controls in Section 3.6 mitigate but do not eliminate this.
  • Managed-service drift. Azure DI is a managed service; any measured result is valid only for the pinned API version and date.
  • Corpus representativeness. The fifty-five cell matrix is broad but cannot represent every real-world invoice template.
  • Ground-truth ambiguity. Some invoice fields are genuinely ambiguous; residual annotation disagreement bounds the achievable accuracy ceiling.

3.8 Where this study sits in the IDP pipeline

An end-to-end intelligent document-processing (IDP) pipeline runs an invoice through a sequence of stages: parse the document, split it into logical documents, classify each document type, extract the structured fields, augment with reference data, validate against business rules, match to purchase orders and receipts, post to the GL, and reason over exceptions. This study scopes specifically to the extract stage — the structured extraction of invoice fields — and does not measure the upstream or downstream stages.

4.1 Per-dimension sample design

Each dimension D1–D14 is provisioned with a target of 5,000 tagged invoice documents (a design parameter), stratified so that every sub-category within the dimension receives a balanced sub-sample. Per-dimension metrics are macro-averaged over strata so that rare but important categories are not outweighed by common ones.

4.2 Aggregate throughput versus time to process

Aggregate throughput (D10) and time to process (D13) are deliberately kept as separate dimensions. D10 measures how many documents a system clears per minute under sustained load — a fleet-level capacity-planning measure reported as a steady-state rate. D13 measures how long a single document takes from submission to a complete result — a per-document latency measure reported at the median and 95th percentile. A platform can show high throughput yet poor tail latency, or the reverse; reporting both avoids a misleading single number.

4.3 Reading the Δ column once measured

  • A Δ with non-overlapping confidence intervals indicates one platform is higher on that dimension's metric; a Δ whose interval includes zero is not a meaningful difference.
  • Overlapping confidence intervals are reported as a tie band; no platform is declared ahead on that dimension.

8. Azure Invoice-Extraction Offerings

“Azure Document Intelligence” is not a single product but a family of distinct offerings, several of which can be applied to invoice extraction in different ways. This section describes the offerings so that a comparison is made against the right one. It is descriptive only and makes no performance claim about any offering.

OfferingWhat it does Pretrained vs. custom Semantic fields vs. raw layout
Document Intelligence &mdash; Prebuilt Invoice A pretrained model that returns semantic invoice fields directly: invoice number and dates, vendor and customer identity, billing and shipping addresses, subtotal, tax, total, and a structured line-item array. Pretrained &mdash; no customer training required. Semantic fields &mdash; returns named invoice fields, not raw layout.
Document Intelligence &mdash; Layout Extracts document structure: text lines and words, selection marks, tables with row and column geometry, and reading order. The structural basis for layout-aware downstream extraction. Pretrained &mdash; structure model, no training required. Raw layout &mdash; returns geometry and tables, not named invoice fields.
Document Intelligence &mdash; Custom extraction (template &amp; neural) A customer-trained model that learns a specific document set. The template variant matches fixed layouts; the neural variant generalises across layout variation within a document type. Custom &mdash; trained on customer-labelled documents. Semantic fields &mdash; returns the customer-defined field schema.
Document Intelligence &mdash; Read / OCR Optical character recognition: detects printed and handwritten text, language, and per-line text with positions across a document. Pretrained &mdash; OCR model, no training required. Raw layout &mdash; returns text and positions, no field semantics.
Document Intelligence &mdash; General document / key-value Extracts generic key-value pairs and entities from an arbitrary document without a document-type-specific model. Pretrained &mdash; general model, no training required. Between raw and semantic &mdash; returns detected key-value pairs, not a fixed invoice schema.
Document Intelligence Studio A web tooling environment for labelling documents, training and testing custom models, and trialling prebuilt models. It is the authoring surface, not an extraction model itself. Tooling &mdash; supports both pretrained trials and custom training. Not applicable &mdash; a labelling and training environment.
Azure OpenAI vision A multimodal large-language-model capability that accepts document images and a prompt, and returns extracted content according to the prompt instructions. Pretrained &mdash; general multimodal model, prompt-driven rather than trained per document type. Prompt-defined &mdash; returns whatever fields the prompt requests; not a fixed invoice schema.

8.1 Choosing the right offering for comparison

The offerings differ along two axes that matter for a fair comparison. The first is pretrained versus custom: a pretrained model works out of the box, while a custom model is trained on customer-labelled documents. The second is semantic fields versus raw layout: some offerings return named invoice fields directly, while others return only text and geometry that a downstream step must interpret. A comparison is only meaningful when each platform is assessed on the offering that matches the capability tier under test — prebuilt against prebuilt, custom against custom, layout against layout — as set out in Section 2.2.

9. Findings & Outlook

9.1 Findings

This edition delivers a framework rather than a verdict. Its findings concern the structure of the comparison and the state of available coverage information; they do not concern measured head-to-head accuracy.

  • A comprehensive and auditable test surface exists. Fourteen non-overlapping dimensions, a fifty-five cell combination matrix and a 404-item taxonomy crosswalk together define an invoice-extraction evaluation surface that is broad, explicit and reproducible.
  • Existing taxonomies under-represent operational dimensions. Confidence calibration, throughput, visual grounding, straight-through-processing rate and per-document latency are absent from both external taxonomies despite being decisive for operational cost. A complete comparison must measure them explicitly.
  • HyperAPI coverage of the complexity space is broad but uneven in depth. HyperAPI's capability-coverage assessment (Section 7) reaches a majority of the 204 invoice-complexity dimensions at least partially, with full coverage concentrated in the structural core and the depth frontier in script handling, locale identifiers and scan quality. This is a scope claim, not a measured result, and it indicates where measured evaluation is most likely to be informative.
  • No comparative performance claim is supported at this stage. Because no head-to-head measurement has been collected, this report makes — and supports — no claim that either platform outperforms the other.

9.2 Outlook

The next edition will move from framework to measurement. It will require: a tagged invoice corpus stratified across the fifty-five cell matrix toward the per-dimension sample target, with double-annotated and adjudicated ground truth; structured cell-level table ground truth for dimension D8; frozen development and test splits with a content manifest; pinned platform references and request configurations; measured metric outputs reported with 95% confidence intervals and sample sizes; a contamination review; and independent review confirming that the measurement followed the methodology in Section 3. Only when those conditions are met will a comparative result cell move from “Not yet measured” to a measured value.

9.3 Recommendations for evaluators

  • Adopt the fourteen-dimension framework, and require any platform comparison to address the operational dimensions D9–D13 that existing taxonomies omit.
  • Treat HyperAPI's capability-coverage assessment as a starting hypothesis, and prioritise measured evaluation in the clusters with the largest not-yet-covered counts.
  • Require that comparative results be reported with confidence intervals and sample sizes, and treat overlapping intervals as a tie band rather than a ranking.

Z.2 The integrity contract

The report is built under an explicit integrity contract whose purpose is to guarantee that no fabricated or self-reported performance number can reach any edition of the document:

  • Every comparative result cell renders an explicit data-status label and never a numeric result. In this edition that label is always “Not yet measured”.
  • No accuracy figure, win-rate or Δ value is emitted into any result column.
  • The 5,000-documents-per-dimension figure is described only as a sample-design parameter, never as a completed measurement.
  • The layout-awareness expectation is treated as a hypothesis to be tested, never stated as a finding.
  • Vendor self-assessed coverage counts (Section 7) are labelled vendor-reported and are never presented as a measured result.

Z.3 The automated integrity guard

The builder enforces the contract in code through the function assert_no_fabricated_numbers(), which runs against the generated HTML before any file is written and aborts the build on violation. The guard performs three checks:

  1. It scans the rendered body (excluding CSS layout widths) for any number followed by a percent sign. The only percentage permitted is a 95% confidence-interval reference, which is a method and not a result; any other percentage aborts the build as a suspected fabricated score.
  2. It confirms that every 95% confidence-interval mention is used only as a method reference, so the figure cannot be a disguised result.
  3. It confirms that the comparative result cells are all present as data-status labels — one label per result cell across the fourteen dimensions — so that no result cell can have been silently replaced with a number.

This guard, and the “no fabricated cell” discipline it enforces, are the reason the report uses an explicit data-status legend rather than blank cells or provisional numbers. The discipline is deliberately kept out of the analytical body of the report: an external reader needs the data-status legend, not the build mechanics.
