SCALE-TO-100 — Pinned path to 100+ models × all tasks × all datasets

Owner: frontier-lab-cto + model-wrangler. Method: .claude/skills/benchmark-factory/SKILL.md. This is the documented, pinned procedure to scale the SEED leaderboard to production. It does NOT run 100+ real inference now (human-locked decision); it specifies exactly how.

1. What already scales unchanged

The SEED harness is the production harness. harness/run.py is task×dataset×model-agnostic: adding models = adding registry entries + adapters; adding tasks = adding harness/tasks.json + harness/metrics.py entries; adding datasets = adding a registered loader. No harness rewrite.

2. Model list shape (target 100+)

harness/models.json, each entry pinned:


{"id","family","adapter","pin":{"kind":"hf-revision|api-version|file-sha256",
 "ref":"<commit/revision/date/sha>"},"cost_per_1k_docs_usd","throughput_docs_per_s",
 "is_hyperbots_ip":bool}

Coverage shape:

  • Open-weights via HF hub (pinned revision=<commit>): small→large

(e.g. 0.5B–70B class), instruction-tuned + base.

  • Hosted-API models (pinned model+version date), keys human-provided.
  • Hyperbots IP: open-source models fine-tuned by Hyperbots, pinned by weight

sha256. Production accuracy cited only "per apis.hyperbots.com", never injected into a measured cell.

3. The HF-hub / API adapter (the only new code to scale models)

Adapter contract is already fixed: pure predict(prompt, decoding) -> text. A scale-up adapter loads the model offline with a pinned revision:


from transformers import pipeline      # pinned in harness/PINS.txt
pipe = pipeline("text-generation", model=ID, revision=COMMIT,
                device_map="auto", torch_dtype="auto")
def predict(prompt, decoding):
    return pipe(prompt, do_sample=decoding["temperature"]>0,
                temperature=decoding["temperature"], top_p=decoding["top_p"],
                max_new_tokens=decoding["max_tokens"])[0]["generated_text"]

Scale-up dependency pins (already staged in harness/PINS.txt, commented until activated): transformers==4.44.2, torch==2.4.1, accelerate==0.34.2, huggingface-hub==0.25.1. Heavy venvs live in /tmp; never committed (.gitignore enforces).

4. Task + dataset scale-out

  • Tasks: walk the 549 implemented catalog rows by family; for each family add

ONE canonical metric in harness/metrics.py + ONE frozen prompt in harness/tasks.json. Reviewed by stats-methodologist.

  • Datasets: register SROIE + CORD (already in registry, status

requires-manual-download); add FUNSD/DocVQA-style sets per task mapping. dataset-curator freezes split + sha256 on first local load. Internal Hyperbots data plugs in via datasets/loaders/internal_adapter.py the moment HYPERBOTS_INTERNAL_DATASET_PATH + HYPERBOTS_INTERNAL_CREDS exist.

5. Cost / throughput plan

  • Budget = Σ over (model,task,dataset) of n_docs / throughput_docs_per_s

(compute) and n_docs/1000 * cost_per_1k_docs_usd (API). Registry carries both fields per model so the total is computed before any large run.

  • Tiering: run small/open models first (cheap, offline, in /tmp venv);

gate expensive API models behind explicit human budget approval.

  • Parallelism: cells are independent → embarrassingly parallel; shard by

(model,task,dataset); each shard writes its own results/*.json.

  • Throughput control + cost ceiling owned by model-wrangler; CTO STOP if a

run would exceed approved budget.

6. Release pipeline (per the skill, every refresh)

curate datasets → register pinned models → harness/run.py shards → contamination/audit.py per (model,dataset) → stats-methodologist validates CIs/tie-bands → harness/repro_check.py (≥1 sampled cell, STOP on mismatch) → leaderboard/build.py regenerates static UI.

7. Where the internal adapter plugs in

datasets/loaders/internal_adapter.py is inert (is_active()==False) until both env vars are set, then it emits the documented record schema and the same harness/contamination/repro pipeline runs over it with zero code change.
