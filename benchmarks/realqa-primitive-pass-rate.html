What this measures

For each of 4,744 generated repos with a canonical schema/expected_fields.json, the realqa runner invokes the repo's primitive (parse / split / classify / extract / detect / validate / reconcile / bundle) against sampled documents and checks whether the SDK response matches the expected schema. A verified pass means the SDK returned the schema. An acceptable-unverified-schema-miss means the SDK returned a response but the field-set did not match. Rows with no schema to compare against are excluded from the denominator.

Distinct from the dataset_95 extract benchmark. The 0.36 micro-F1 on dataset_95 measures field-extraction accuracy on real-world invoices — a different axis. The 93.1% above measures SDK-functional correctness across the 8-primitive surface. Both are honest. Both refer to different things.

Methodology

  • Endpoint: live apis.hyperbots.com (production, not a fork or a staging mirror).
  • Sampling: stratified by document-type-hint (invoice / receipt / PO / bank_statement / credit_note / remittance_advice / unknown), weighted inverse-by-utilization so under-sampled docs catch up over iterations.
  • Schema source: generated/<slug>/schema/expected_fields.json — derived from each repo's rules.py in PR #11 (2026-05-26).
  • Pass criterion: SDK response contains the expected schema keys (no field-set mismatch).
  • Excluded verdicts: skipped-sdk-surface (primitive not implemented in pinned SDK), blocked-sdkbug-flap (known upstream flap, recorded but not counted), error-nonflap (recorded separately).

Reproducibility

Source: hyperapi-repo-factory/realqa/scale/PROGRESS.jsonl. To re-count: filter rows where verdict ∈ {pass, acceptable-unverified-schema-miss} — the pass-rate is count(verdict='pass') ÷ that subset. As of 2026-05-27 20:00 IST the ledger is 339,632 rows, growing at ~210/min as the runner continues to sample.

What this is NOT

  • NOT a comparative claim against Gemini / Azure DI / GPT — this measures only HyperAPI primitives against their own expected schemas.
  • NOT a field-extraction-accuracy claim — see the dataset_95 row (0.36 F1) in the consolidated leaderboard.
  • NOT a latency claim — see ocr-latency rows.
  • NOT a customer-cited claim — no customer names are referenced (partner-PII gate v1).
